async/await w kontekście testów

Dlaczego asynchroniczność jest kluczowa w automatyzacji

Zaczynamy od async/await, bo Playwright wymaga tej składni w każdym teście. Cypress ukrywa asynchroniczność za łańcuchami komend, ale warto rozumieć co się dzieje pod spodem.

Dlaczego asynchroniczność?

W świecie testów E2E prawie wszystko trwa jakiś czas:

  • Ładowanie strony
  • Kliknięcie przycisku i oczekiwanie na odpowiedź serwera
  • Pojawienie się elementu na stronie
  • Zapytanie do API
  • Zapis/odczyt pliku

JavaScript nie może "czekać" - musi być poinformowany, kiedy operacja się skończy.

Wytłumacz analogia - wyobraź sobie, że zamawiasz jedzenie na dowóz. Nie stoisz przy drzwiach i czekasz - robisz inne rzeczy, a kiedy kurier przyjdzie, reagujesz. Tak samo działa JS.

Kod synchroniczny vs asynchroniczny

Synchroniczny


const a = 1 + 2;
const b = a * 3;
console.log(b); // 9
// kazda linia czeka na poprzednia

Asynchroniczny


// to trwa - musimy "poczekac"
const response = await fetch('/api');
const data = await response.json();
console.log(data);
Zwróć uwagę na słowo kluczowe await - to ono mówi "poczekaj na wynik tej operacji zanim przejdziesz dalej". Bez await dostaniemy obiekt Promise zamiast prawdziwych danych.

Callback - funkcja zwrotna

Najprostszy sposób obsługi asynchroniczności - przekazanie funkcji do wywołania "później":


// przyklad z zycia - setTimeout
setTimeout(() => {
  console.log('To pojawi sie po 2 sekundach');
}, 2000);

console.log('To pojawi sie natychmiast');
    
To pojawi się natychmiast
// ...po 2 sekundach...
To pojawi się po 2 sekundach
Callbacki to najprostsza forma asynchroniczności. Problem pojawia się kiedy mamy wiele zagnieżdżonych callbacków - tzw. "callback hell". Dlatego wymyślono Promise i async/await.

Promise - obietnica

Promise to obiekt reprezentujący przyszły wynik operacji:


// Promise moze byc w jednym z 3 stanow:
// - pending   (oczekujacy)
// - fulfilled (spelniony - sukces)
// - rejected  (odrzucony - blad)

const promise = fetch('https://api.example.com/data');

promise
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error('Blad:', error));
Promise to "pudełko na przyszły wynik". Metoda then() obsługuje sukces, catch() obsługuje błąd. Można je łańcuchowo łączyć. Ale my będziemy używać ładniejszej składni - async/await.

async/await - czytelna skladnia

async/await to "cukier składniowy" nad Promise - ten sam mechanizm, ładniejszy zapis:


// funkcja oznaczona jako async
async function fetchUserData(userId) {
  // await "czeka" na wynik Promise
  const response = await fetch(`/api/users/${userId}`);
  const user = await response.json();
  return user;
}

// uzycie
const user = await fetchUserData(42);
console.log(user.name);

Zasada

await może być użyte tylko wewnątrz funkcji async

Słowo kluczowe async przed funkcją mówi JS, że ta funkcja jest asynchroniczna i będzie zwracać Promise. Słowo kluczowe await wewnątrz takiej funkcji "zatrzymuje" wykonanie do momentu rozwiązania Promise. To tak jakby pisać kod synchroniczny, ale z magicznym czekaniem na wyniki.

Obsługa błędów - try/catch


async function fetchProduct(productId) {
  try {
    const response = await fetch(`/api/products/${productId}`);

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }

    const product = await response.json();
    return product;

  } catch (error) {
    console.error('Nie udalo sie pobrac produktu:', error.message);
    return null;
  }
}
try/catch to standardowy sposób obsługi błędów w async/await. Bez try/catch błąd w await "poleci" w górę i może zatrzymać cały test. W testach E2E błędy są często oczekiwane - np. testujemy, że strona zwraca 404.

Cypress - kolejkowanie komend

Cypress ukrywa asynchroniczność za łańcuchami komend:


it('should login successfully', () => {
  cy.visit('/login');
  cy.get('[data-testid="email"]').type('user@test.pl');
  cy.get('[data-testid="password"]').type('Test1234');
  cy.get('[data-testid="login-btn"]').click();
  cy.url().should('include', '/dashboard');
});

Zauważ

Cypress kolejkuje komendy automatycznie - nie potrzebujemy async/await

Cypress ma własny system kolejkowania komend. Nie używamy async/await - komendy są dodawane do kolejki i wykonywane po kolei. To prostsze do napisania, ale może być mylące - ten kod NIE wykonuje się synchronicznie, choć tak wygląda. Cypress sam zarządza czekaniem.

A w Playwright - async/await

Playwright wymaga jawnego uzycia async/await:


test('should login successfully', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[data-testid="email"]', 'user@test.pl');
  await page.fill('[data-testid="password"]', 'Test1234');
  await page.click('[data-testid="login-btn"]');
  await expect(page).toHaveURL('/dashboard');
});

Roznica

Każda interakcja z przeglądarka wymaga await - bo trwa jakiś czas

To jest prawdziwy test Playwright. Zwróć uwagę: async w deklaracji funkcji testowej, await przed każda komenda. Gdybyśmy pominęli await, komendy nie wykonałyby się po kolei i test by się nie powiódł. Playwright automatycznie czeka na elementy - ale await jest potrzebne do prawidłowej kolejności operacji.

Cypress vs Playwright - podsumowanie różnic

Cecha Cypress Playwright
async/await Nie używane Wymagane
Oczekiwanie na elementy Auto-retry Auto-wait + await
Wiele zakładek Ograniczone Tak
Język testów JS/TS JS/TS/Python/C#/Java
Przeglądarki Chrome, Firefox, Edge Chromium, Firefox, WebKit
Nie chodzi o to, który jest lepszy - oba mają swoje zastosowania. Ważne jest, żeby rozumieć różnice w modelu działania. W Rossmann możecie używać obu narzędzi w różnych projektach.

Czekanie na elementy w testach

Cypress


// automatycznie czeka (4s)
cy.get('.modal')
  .should('be.visible');

// czeka az tekst sie pojawi
cy.get('.alert')
  .should('have.text', 'Sukces!');

// zmiana timeout
cy.get('.loader',
  { timeout: 10000 })
  .should('exist');

Playwright


// czeka az element bedzie widoczny
await page.locator('.modal')
  .waitFor();

// czeka az tekst sie pojawi
await expect(
  page.locator('.alert')
).toHaveText('Sukces!');

// timeout - max czas czekania
await page.locator('.loader')
  .waitFor({ timeout: 10000 });
Oba narzędzia mają wbudowane mechanizmy czekania. Nie używajcie sleep/wait z ustalonym czasem - to prowadzi do wolnych i niestabilnych testów. Zamiast tego czekajcie na konkretne warunki.

Ćwiczenie - async/await

Napisz asynchroniczna funkcje getUser, która:

  1. Pobiera dane z https://jsonplaceholder.typicode.com/users/1
  2. Parsuje odpowiedź jako JSON
  3. Zwraca obiekt z polami name i email
  4. Obsługuje błędy za pomocą try/catch

async function getUser() {
  try {
    const response = await fetch(
      'https://jsonplaceholder.typicode.com/users/1'
    );
    const data = await response.json();
    return { name: data.name, email: data.email };
  } catch (error) {
    console.error('Blad pobierania:', error.message);
    return null;
  }
}
Daj uczestnikom 10-15 minut na to ćwiczenie. Mogą używać konsoli przeglądarki. Podpowiedź: fetch() zwraca Promise, response.json() też zwraca Promise - oba potrzebują await.

Selektory w testach E2E

Jak wybierać elementy stabilnie i czytelnie

To jest kluczowa sekcja dzisiejszego dnia. Uczestnicy prosili o ten temat - selekcja elementów to fundament każdego testu E2E. Źle selektory = niestabilne testy.

Przypomnienie - selektory CSS

Selektor Przykład Opis
Element button Wszystkie przyciski
Klasa .btn-primary Element z klasa
ID #login-form Element z ID
Atrybut [type="submit"] Element z atrybutem
Potomek .form input input wewnątrz .form
Dziecko .form > input Bezpośrednie dziecko
Szybkie przypomnienie selektorów CSS. Większość uczestników zna te selektory z pracy testerskiej. Ważne jest, żeby rozumieć różnice między potomkiem a bezpośrednim dzieckiem.

Hierarchia strategii selekcji

Od najlepszej do najgorszej:

  1. data-testid - najbardziej stabilny
  2. Role-based (getByRole) - semantyczny
  3. Tekst (getByText) - czytelny, ale kruchy
  4. CSS selektory (id, klasa) - kruchy
  5. XPath - unikaj
To jest kluczowa zasada. data-testid to atrybut dodany specjalnie do celów testowych - nie zmienia się przy zmianach CSS, nie zależy od tekstu (który może się zmienić w tłumaczeniu). Role-based selektory są dobre bo testują dostępność. XPath jest kruchy i nieczytelny.

data-testid - zloty standard


<!-- HTML w aplikacji -->
<button
  class="btn btn-primary mx-2 rounded-lg"
  data-testid="add-to-cart-btn"
>
  Dodaj do koszyka
</button>
    

// Cypress
cy.get('[data-testid="add-to-cart-btn"]').click();

// Playwright
await page.locator('[data-testid="add-to-cart-btn"]').click();

Dlaczego data-testid?

Nie zmienia się przy zmianach CSS, nie zależy od tekstu, jest dedykowany do testów

data-testid to atrybut HTML, który nie wpływa na wyglad ani działanie strony. Służy wyłącznie do identyfikacji elementów w testach. Deweloperzy dodają go do kluczowych elementów. Konwencja: kebab-case, opisowa nazwa.

Konwencje nazewnictwa data-testid


<!-- Dobre nazwy -->
<input data-testid="login-email-input" />
<button data-testid="submit-order-btn" />
<div data-testid="product-card" />
<span data-testid="cart-item-count" />

<!-- Zle nazwy -->
<input data-testid="input1" />
<button data-testid="btn" />
<div data-testid="div123" />

Uwaga

Nazwy powinny opisywać co to jest, nie jak wygląda

Dobre nazwy pomagają zrozumieć test bez patrzenia na stronę. "login-email-input" mówi dokładnie co to za element. "input1" nie mówi nic. Ustalcie konwencje z deweloperami w zespole.

Selektory oparte na rolach


// Cypress z Testing Library
cy.findByRole('button', { name: 'Zaloguj' }).click();
cy.findByRole('textbox', { name: 'Email' }).type('a@b.pl');

// Playwright - getByRole
await page.getByRole('button', { name: 'Zaloguj' }).click();
await page.getByRole('textbox', { name: 'Email' }).fill('a@b.pl');
await page.getByRole('heading', { name: 'Dashboard' }).isVisible();
await page.getByRole('link', { name: 'Produkty' }).click();

Bonus

Selektory oparte na rolach testują równocześnie dostępność (a11y) aplikacji

Role-based selektory to podejście Playwright i Testing Library. Używają rol ARIA - button, textbox, heading, link itd. Nazwa pochodzi z aria-label, tekstu wewnątrz elementu, lub atrybutu label. Dobre podejście, bo testuje też czy strona jest dostępna.

Selektory oparte na tekście


// Cypress
cy.contains('Dodaj do koszyka').click();
cy.contains('button', 'Zaloguj').click();

// Playwright
await page.getByText('Dodaj do koszyka').click();
await page.getByLabel('Adres email').fill('a@b.pl');
await page.getByPlaceholder('Wyszukaj produkt...').fill('szampon');

Uwaga

Tekst może się zmienić przy tłumaczeniu lub redesignie - używaj ostrożnie

Selektory tekstowe są bardzo czytelne - od razu widać co test robi. Ale mają wady: jeśli ktoś zmieni tekst przycisku z "Zaloguj" na "Zaloguj się", test się zepsuje. W projektach wielojęzycznych tekst zmienia się często. Dobre dla elementów, które raczej nie zmienia tekstu.

Problem: dynamiczne klasy CSS

Współczesne aplikację generują nazwy klas automatycznie:


<!-- CSS Modules -->
<button class="Button_primary__x7k2m9">Kup</button>

<!-- Styled Components -->
<button class="sc-bdnxRM jRqWYa">Kup</button>

<!-- Tailwind + build hash -->
<button class="bg-blue-500 hover:bg-blue-700 text-white
  font-bold py-2 px-4 rounded">Kup</button>

Te klasy mogą się zmienić przy każdym buildzie!

To jest częsty problem - klasy CSS są generowane przez narzędzia budujące (Webpack, Vite) i zmieniają się między deployami. Test oparty na klasie "Button_primary__x7k2m9" zepsuje się przy następnym buildzie. Dlatego data-testid jest tak ważne.

Dzisiejszy build vs jutrzejszy

Build #142


<button class="btn-x7k2m9">
  Kup teraz
</button>
        

// Ten selektor dziala dzis
cy.get('.btn-x7k2m9').click();

Build #143


<button class="btn-a3p1q8">
  Kup teraz
</button>
        

// Ten sam selektor - FAIL!
cy.get('.btn-x7k2m9').click();

Rozwiązanie: [data-testid="buy-now-btn"] - stabilne między buildami

Namaluj ten scenariusz jasno - deweloper nic nie zmienił w logice, tylko zbudował ponownie aplikację i klasy CSS się zmieniły. data-testid nie zmienia się między buildami.

Problem: biblioteki komponentów

Material UI, Ant Design - generują złożony DOM:


<!-- Prosty HTML button -->
<button>Wyslij</button>

<!-- Material UI Button - wygenerowany DOM -->
<button class="MuiButtonBase-root MuiButton-root
  MuiButton-contained MuiButton-containedPrimary
  MuiButton-sizeMedium css-1hw9j7s">
  <span class="MuiButton-startIcon
    MuiButton-iconSizeMedium css-1d6wzja">
    <svg>...</svg>
  </span>
  <span class="MuiButton-label">Wyslij</span>
  <span class="MuiTouchRipple-root css-w0pj6f"></span>
</button>
Material UI generuje dużo dodatkowego kodu HTML. Klasy MUI mogą się zmienić między wersjami biblioteki. Nigdy nie opierajcie selektorów na wewnętrznych klasach biblioteki komponentów.

Rozwiązania


<!-- 1. data-testid na wrapperze -->
<button data-testid="submit-btn"
  class="MuiButton-root ...">
  Wyslij
</button>
    

// 2. Cypress - contains (szuka tekstu)
cy.contains('button', 'Wyslij').click();

// 3. Playwright - getByRole (ignoruje wewnetrzna strukture)
await page.getByRole('button', { name: 'Wyslij' }).click();

// 4. Playwright - getByText
await page.getByText('Wyslij').click();
Najlepsze rozwiązanie to data-testid, ale jeśli nie macie możliwości dodać atrybutów do kodu (np. aplikacja zewnętrzna), używajcie getByRole lub getByText. Te selektory nie zależą od wewnętrznej struktury DOM generowanej przez biblioteki.

Praktyka - jak wybrać ten element?


<form id="login-form">
  <div class="form-group mb-3">
    <label for="email">Email</label>
    <input
      type="email"
      id="email"
      class="form-control"
      data-testid="login-email"
      placeholder="Wpisz email"
    />
  </div>
</form>

Jak najlepiej wybrać to pole email?

  • [data-testid="login-email"] - najlepszy
  • getByRole('textbox', { name: 'Email' }) - również dobry
  • #email - akceptowalny (ID jest stabilne)
  • .form-control - zly (wiele elementów z ta klasa)
Interaktywne ćwiczenie - zapytaj uczestników zanim pokażesz odpowiedzi (fragmenty). Wytłumacz dlaczego każda opcja jest lepsza lub gorsza.

Praktyka - lista produktów


<div class="product-grid">
  <div class="card" data-testid="product-card">
    <img src="shampoo.jpg" alt="Szampon" />
    <h3 data-testid="product-name">Szampon Isana</h3>
    <span data-testid="product-price">12.99 zl</span>
    <button data-testid="add-to-cart">Dodaj</button>
  </div>
  <div class="card" data-testid="product-card">
    <!-- kolejny produkt -->
  </div>
</div>

Jak dodać do koszyka pierwszy produkt?


// Cypress - .first()
cy.get('[data-testid="add-to-cart"]').first().click();

// Playwright - .first()
await page.locator('[data-testid="add-to-cart"]').first().click();
Częste pytanie - co robić jeśli jest wiele elementów z tym samym data-testid? Używamy .first(), .last(), .nth(n) lub zawężamy kontekst do rodzica. Można też użyć bardziej specyficznego data-testid, np. data-testid="add-to-cart-szampon-isana".

Zawężanie kontekstu selektora


// Cypress - within
cy.get('[data-testid="product-card"]').first().within(() => {
  cy.get('[data-testid="add-to-cart"]').click();
});

// Cypress - contains wewnatrz
cy.contains('[data-testid="product-card"]', 'Szampon Isana')
  .find('[data-testid="add-to-cart"]').click();
    

// Playwright - szukaj wewnatrz konkretnego elementu
const firstCard = page.locator('[data-testid="product-card"]').first();
await firstCard.locator('[data-testid="add-to-cart"]').click();

// Playwright - filter po tekscie
await page.locator('[data-testid="product-card"]')
  .filter({ hasText: 'Szampon Isana' })
  .locator('[data-testid="add-to-cart"]')
  .click();
Zawężanie kontekstu to potężna technika. Zamiast szukać globalnie na całej stronie, najpierw znajdź rodzica (np. karte produktu), a potem szukaj wewnątrz niego. To sprawia, że selektory są precyzyjne i odporne na zmiany.

Cypress - przegląd selektorów


// CSS selector
cy.get('[data-testid="email"]')
cy.get('#email')
cy.get('.btn-primary')

// Szukanie po tekscie
cy.contains('Zaloguj')
cy.contains('button', 'Zaloguj')

// Kombinowanie
cy.get('[data-testid="product-card"]').first()
cy.get('[data-testid="product-card"]').eq(2) // trzeci element

// Wewnatrz elementu
cy.get('.modal').find('[data-testid="close-btn"]')
cy.get('.form').within(() => {
  cy.get('input').first().type('tekst');
});
Cypress jest prostszy - głównie cy.get() z selektorami CSS i cy.contains() do tekstu. eq() to odpowiednik nth() w Playwright. within() ogranicza kontekst wyszukiwania.

Playwright - przegląd lokatorów


// CSS selector
page.locator('[data-testid="email"]')
page.locator('#email')
page.locator('.btn-primary')

// Wbudowane lokatory
page.getByRole('button', { name: 'Zaloguj' })
page.getByText('Witaj, Anna')
page.getByLabel('Email')
page.getByPlaceholder('Szukaj...')
page.getByTestId('submit-btn')  // skrot dla data-testid

// Filtrowanie
page.locator('button').filter({ hasText: 'Zapisz' })
page.locator('.card').filter({ has: page.locator('.badge') })
To jest ściąga z najczęściej używanych lokatorów Playwright. getByTestId to skrot - domyślnie szuka atrybutu data-testid, ale można to skonfigurować w playwright.config.ts. Warto wydrukować te ściągę uczestnikom.

Checklist stabilności selektorów

  • Czy selektor przetrwa zmianę styli CSS?
  • Czy selektor przetrwa zmianę tekstu na stronie?
  • Czy selektor jest unikalny na stronie? (sprawdź w DevTools)
  • Czy selektor jest czytelny dla innego testera?
  • Czy selektor zależy od głębokość struktury DOM?

Przed każdym selektorem zadaj sobie pytanie: "Czy to się zepsuje przy następnym deployie?"

Ten checklist powinien być w głowie każdego testera piszącego testy automatyczne. Jeśli selektor jest długi i zagłębia się w wiele poziomów DOM - to czerwona flaga.

Ćwiczenie - napisz najlepszy selektor

Dla każdego fragmentu HTML napisz optymalny selektor:

  Koszyk (3)
</a>

<!-- 2 -->
<input type="search" placeholder="Szukaj produktu..."

Odpowiedzi: 1) [data-testid="nav-cart"] - najlepsza opcja. 2) Brak data-testid - getByPlaceholder('Szukaj produktu...') lub getByRole('searchbox'). Poproś deweloperów o dodanie data-testid. 3) Klasa dynamiczna! Użyj getByText('Zamowienie zostalo zlozone pomyslnie!') lub poproś o data-testid.

Organizacja kodu, Page Object Model i klasy

Teraz pokażemy jak organizować projekt testowy. To ważne od samego początku - dobrze zorganizowany projekt jest łatwiejszy do utrzymania i rozbudowy.

Typowa struktura projektu


tests/

Bez POM

Page Object Model to wzorzec projektowy. Zamiast powielać selektory w wielu testach, tworzymy klasę, która reprezentuje stronę i zawiera wszystkie selektory i akcje. Jeśli selektor się zmieni - aktualizujemy w jednym miejscu.

Klasy w JavaScript

Klasa to "szablon" do tworzenia obiektów z tymi samymi właściwościami i metodami:


class Animal {
  constructor(name, sound) {
    this.name = name;
    this.sound = sound;
  }

  speak() {
    return `${this.name} mowi: ${this.sound}!`;
  }
}

const dog = new Animal('Burek', 'Hau');
console.log(dog.speak()); // "Burek mowi: Hau!"

const cat = new Animal('Mruczek', 'Miau');
console.log(cat.speak()); // "Mruczek mowi: Miau!"
Klasy to nowy temat - nie omawialiśmy ich wcześniej bo czekaliśmy na kontekst POM. Constructor to specjalna metoda, która uruchamia się przy tworzeniu obiektu (new). this odnosi się do konkretnej instancji obiektu.

Klasy - kluczowe elementy


class Product {
  // constructor - ustawia poczatkowe wartosci
  constructor(name, price) {
    this.name = name;
    this.price = price;
    this.inCart = false;
  }

  // metoda - funkcja nalezaca do klasy
  addToCart() {
    this.inCart = true;
    return `${this.name} dodano do koszyka`;
  }

  // metoda z parametrem
  applyDiscount(percent) {
    this.price = this.price * (1 - percent / 100);
    return this.price;
  }
}

const shampoo = new Product('Szampon Isana', 12.99);
shampoo.applyDiscount(10); // 11.691
Pokazujemy klasę z metodami. this to referencja do bieżącego obiektu. Metody mogą modyfikować stan obiektu i zwracać wartości. Za chwile zobaczymy jak to wygląda w Page Object Model.

Page Object - Cypress (bez klas)

W Cypress częściej używamy obiektów z funkcjami zamiast klas:


// pages/loginPage.js
export const loginPage = {
  visit: () => cy.visit('/login'),

  fillEmail: (email) =>
    cy.get('[data-testid="email"]').type(email),

  fillPassword: (pwd) =>
    cy.get('[data-testid="password"]').type(pwd),

  submit: () =>
    cy.get('[data-testid="login-btn"]').click(),

  login: (email, pwd) => {
    loginPage.visit();
    loginPage.fillEmail(email);
    loginPage.fillPassword(pwd);
    loginPage.submit();
  }
};
Cypress nie wymaga klas - obiekty z funkcjami są prostsze i wystarczające. Cypress ma swoj model asynchroniczności - nie używamy async/await. Ten sam wzorzec - selektory w jednym miejscu, metody do akcji.

Użycie Page Object w Cypress


// specs/login.cy.js
import { loginPage } from '../pages/loginPage';

describe('Login', () => {

  it('powinien zalogowac uzytkownika', () => {
    loginPage.login('user@test.pl', 'Test1234');
    cy.url().should('include', '/dashboard');
  });

  it('powinien pokazac blad dla zlego hasla', () => {
    loginPage.login('user@test.pl', 'wrong');
    cy.get('[data-testid="error-msg"]')
      .should('contain', 'Nieprawidlowe dane');
  });

});
Test mówi CO robimy, a nie JAK. Page Object ukrywa szczegóły implementacji (selektory, kroki).

W Playwright (który wymaga async/await) wygląda to tak:


// pages/LoginPage.ts
class LoginPage {
  constructor(page) {
    this.page = page;
    this.emailInput = page.locator('[data-testid="email"]');
    this.passwordInput = page.locator('[data-testid="password"]');
    this.loginButton = page.locator('[data-testid="login-btn"]');
    this.errorMessage = page.locator('[data-testid="error-msg"]');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email, password) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }

  async getError() {
    return await this.errorMessage.textContent();
  }
}
To jest kompletny Page Object dla strony logowania w Playwright. Zwróć uwagę: selektory są w konstruktorze, akcje są metodami async. W teście używamy this.emailInput zamiast powtarzać selektor.

Użycie Page Object w teście Playwright


// specs/login.spec.ts
import { LoginPage } from '../pages/LoginPage';

test.describe('Login', () => {

  test('powinien zalogowac uzytkownika', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login('user@test.pl', 'Test1234');
    await expect(page).toHaveURL('/dashboard');
  });

  test('powinien pokazac blad dla zlego hasla', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login('user@test.pl', 'wrong');
    const error = await loginPage.getError();
    expect(error).toContain('Nieprawidlowe dane');
  });

});
Zauważ jak czytelne są testy z Page Object. Nie ma selektorów w testach - test czyta się jak instrukcja: "idz na stronę logowania, zaloguj się, sprawdź URL". Jeśli selektor się zmieni - aktualizujesz LoginPage, a nie każdy test.

Element Konwencja Przykład

Plik testowy (CY) nazwa.cy.js login.cy.js

Co testować automatycznie?

Strategia wyboru testów E2E

Ważna sekcja - nie chodzi o to żeby automatyzować wszystko. Chodzi o to, żeby automatyzować mądrze. Źle dobrane testy kosztują więcej niż dają.

Piramida testów


            /\
           /  \         E2E (testy end-to-end)
          / E2E\        - malo, ale kluczowe
         /------\       - wolne, drogie w utrzymaniu
        / Integ- \
       / racyjne  \     Integracyjne
      /------------\    - srednia ilosc
     /    Testy     \
    /  jednostkowe   \  Unit - duzo, szybkie, tanie
   /------------------\

Testy E2E są na szczycie - powinno ich być najmniej, ale obejmują najważniejsze scenariusze.

Piramida testów mówi: większość testów powinna być jednostkowych (szybkie, tanie), mniej integracyjnych, a najmniej E2E. Testy E2E są wolne, kruche i kosztowne w utrzymaniu. Ale są niezbędne do testowania całych przepływów.

Co przynosi wartość w E2E?

  • Happy paths - główne ścieżki użytkownika (login, zakup, rejestracja)
  • Krytyczne przepływy biznesowe - koszyk, płatność, zamówienie
  • Obszary podatne na regresje - często psujące się funkcjonalności
  • Smoke testy - czy kluczowe strony w ogóle się ładują?
  • Cross-browser - czy działa na Chrome, Firefox, Safari?
Happy path to scenariusz, w którym wszystko idzie dobrze - uzytkownik loguje się, dodaje produkt, płaci i dostaje potwierdzenie. To powinno być przetestowane jako pierwsze. Krytyczne przepływy to te, które bezpośrednio wpływają na przychód firmy.

Co często NIE przynosi wartości?

  • Testowanie każdej walidacji formularza na poziomie E2E
  • Testowanie pixel-perfect wyglądu (chyba że używasz visual regression)
  • Testowanie wewnętrznej logiki komponentów zewnętrznych
  • Testowanie każdego edge case na poziomie E2E
  • Testowanie funkcji, które zmieniają się co sprint

Zasada

Jeśli coś można przetestować taniej (unit test, test API) - zrob to na niższym poziomie

Częsty błąd początkujących - automatyzacja wszystkiego na poziomie E2E. Test E2E dla walidacji "hasło musi mieć min 8 znaków" to przesada - to powinien być unit test. E2E powinno testować: "czy logowanie działa? czy błąd się wyświetla przy złym haśle?"

Przykład: zakres testów logowania

TAK - E2E

  • Poprawne logowanie
  • Logowanie ze złym hasłem
  • Przekierowanie po logowaniu
  • Wylogowanie

MOŻE

  • "Zapamiętaj mnie"
  • Toggle widoczności hasła
  • Logowanie przez Google

Przykład: zakres testów logowania

NIE - na poziomie E2E

  • Walidacja długości hasła (min 8 znaków)
  • Walidacja formatu email
  • Walidacja znaków specjalnych
  • Każda kombinacja błędnych danych

Zasada

Te walidacje testuj na poziomie unit lub integracyjnym - szybciej i stabilniej.

Konkretny przykład - logowanie. 4 testy E2E wystarczają. Szczegółowe walidacje testujemy na poziomie unit (walidator emaila) lub integracyjnym (formularz). "Może" zależy od kontekstu - jeśli "zapamiętaj mnie" jest kluczowe dla biznesu, testujemy.

Flaky tests - niestabilne testy

Test, który raz przechodzi, raz nie - bez zmian w kodzie:

  • Przyczyna: czekanie na stały czas (cy.wait(5000))
  • Przyczyna: zależność od danych w bazie
  • Przyczyna: zależność od kolejności testów
  • Przyczyna: animacje i przejścia CSS
  • Przyczyna: niestabilne selektory
  • Przyczyna: timeouty przy wolnym CI/CD
Flaky testy to plaga automatyzacji. Zespol traci zaufanie do testów - "a, to się znowu losowo zepsuło, puścimy jeszcze raz". To prowadzi do ignorowania prawdziwych błędów. Ważne: każdy flaky test należy natychmiast naprawić lub wyłączyć.

Zapobieganie niestabilności

Źle


// Stale czekanie
cy.wait(5000);
cy.get('.btn').click();

// Zalezy od danych
cy.get('.product')
  .should('have.length', 42);

// Selektor na klasie CSS
cy.get('.btn-x7k2').click();

Dobrze


// Czekaj na warunek
cy.get('.btn')
  .should('be.visible')
  .click();

// Sprawdz warunek ogolny
cy.get('.product')
  .should('have.length.gt', 0);

// data-testid
cy.get('[data-testid="buy"]')
  .click();
Kluczowe zasady: nigdy cy.wait() z ustalonym czasem, zawsze czekaj na warunek. Nie zależaj od konkretnej ilości danych. Używaj stabilnych selektorów. Każdy test powinien być niezależny - nie polegaj na wyniku poprzedniego testu.

Każdy test musi być niezależny

Testy nie mogą zależeć od siebie!


// ZLE - test 2 zalezy od testu 1
test('should create product', async ({ page }) => {
  // tworzy produkt "Test Shampoo"
});
test('should edit product', async ({ page }) => {
  // szuka "Test Shampoo" - FAIL jesli test 1 nie przeszedl!
});

// DOBRZE - kazdy test jest niezalezny
test('should edit product', async ({ page }) => {
  // najpierw tworzy produkt przez API (setup)
  await apiHelper.createProduct({ name: 'Test Shampoo' });
  // potem edytuje
  await page.goto('/products');
  // ...
});
Jeśli test A tworzy dane, a test B je używa - to źle. Jeśli test A padnie, test B też padnie, choć nie ma żadnego błędu w edycji produktu. Każdy test powinien przygotować swoje dane (np. przez API) i posprzątać po sobie.

Które testy są warte automatyzacji E2E?

  1. Proces składania zamówienia od koszyka do potwierdzenia
  2. Walidacja, że email musi zawierać znak @
  3. Sprawdzenie, że strona główna się ładuje
  4. Testowanie 50 różnych kombinacji danych w formularzu
  5. Logowanie i sprawdzenie, że imię użytkownika jest widoczne

Odpowiedzi

  • 1 - TAK - kluczowy przepływ biznesowy
  • 2 - NIE - unit test
  • 3 - TAK - smoke test
  • 4 - NIE - za dużo, użyj testów na niższym poziomie
  • 5 - TAK - happy path logowania
Interaktywne ćwiczenie - zapytaj uczestników zanim pokażesz odpowiedzi. Dyskusja o każdym przypadku. Punkt 4 to częsty błąd - testowanie danych powinno być na poziomie unit, a E2E testuje przepływ "czy formularz w ogóle działa".

Scenariusze z rossmann-test-app - co automatyzujesz?

  1. Wyszukiwanie produktu i sprawdzenie wyników
  2. Sprawdzenie, że tło przycisku ma dokładnie kolor #E3000F
  3. Dodanie produktu do koszyka i sprawdzenie licznika
  4. Sprawdzenie czy tooltip pojawia się po 300ms
  5. Rejestracja nowego konta i potwierdzenie emailem

Odpowiedzi

  • 1 - TAK - kluczowa funkcjonalność
  • 2 - NIE - visual regression tool
  • 3 - TAK - kluczowy przepływ
  • 4 - NIE - zbyt kruchy i niskopoziomowy
  • 5 - ZALEŻY - jeśli można zautomatyzować email
Kontynuacja dyskusji. Punkt 2 - kolory testujemy narzędziem do visual regression (Percy, Chromatic), nie ręcznym sprawdzaniem wartości CSS. Punkt 4 - timing-based testy są niestabilne. Punkt 5 - email verification można obejść przez API lub Mailosaur/Mailtrap.

Użycie .env w testach


// Cypress - env ustawiane w cypress.config.js lub cypress.env.json
it('login', () => {
  cy.visit(Cypress.env('BASE_URL') + '/login');
  cy.get('#email').type(Cypress.env('TEST_USER_EMAIL'));
  cy.get('#password').type(Cypress.env('TEST_USER_PASSWORD'));
});
    

// Playwright - uzycie process.env
test('login', async ({ page }) => {
  await page.goto(process.env.BASE_URL + '/login');
  await page.fill('#email', process.env.TEST_USER_EMAIL);
  await page.fill('#password', process.env.TEST_USER_PASSWORD);
});
process.env to obiekt w Node.js, który daje dostęp do zmiennych środowiskowych. Cypress ma własny system env - Cypress.env().

Fixtures - dane testowe w plikach


// fixtures/users.json
{

    "email": "tester@rossmann.pl",

    "name": "Anna Testerska"

// Cypress - wbudowana obsluga fixtures
it('should login with fixture data', () => {
  cy.fixture('users.json').then((users) => {
    loginPage.login(
      users.validUser.email,
      users.validUser.password
    );
  });
});
Fixtures to pliki JSON z danymi testowymi. Trzymamy je oddzielnie od testów - łatwiej aktualizować dane bez dotykania logiki testów. W Cypress fixtures są wbudowane i ładowane przez cy.fixture().

Fixtures - więcej przykładów


// Cypress - alternatywnie z beforeEach
describe('Login', () => {
  beforeEach(function() {
    cy.fixture('users.json').as('users');
  });

  it('should login', function() {
    loginPage.login(
      this.users.validUser.email,
      this.users.validUser.password
    );
  });
});

// Playwright - import fixture
import users from '../fixtures/users.json';

test('login as valid user', async ({ page }) => {
  await loginPage.login(
    users.validUser.email,
    users.validUser.password
  );
});
Cypress ma wbudowaną obsługę fixtures - plik musi być w cypress/fixtures/. cy.fixture() automatycznie parsuje JSON. Można użyć aliasu (.as) do przechowania danych. Uwaga na function() vs arrow function - this nie działa z arrow functions w Mocha. W Playwright importujemy JSON bezpośrednio.

Strategie zarzadzania danymi

Strategia Kiedy używać Przykład
Statyczne fixture Proste, stałe dane users.json
Faker.js Unikalne dane per test losowy email, imię
API seeding Przygotowanie stanu Utwórz usera przez API
DB seeding Złożone scenariusze Pełna baza testowa 

// Faker.js - generowanie unikalnych danych
import { faker } from '@faker-js/faker';

const testUser = {
  name: faker.person.fullName(),
  email: faker.internet.email(),
  password: faker.internet.password({ length: 12 })
};
Różne strategie pasują do różnych sytuacji. Statyczne fixture dla prostych testów, faker.js kiedy potrzebujemy unikatowych danych (np. rejestracja - każdy test nowy email), API seeding kiedy musimy przygotować stan (np. utworzyć użytkownika przed testem logowania).

Realne dane klientów w testach = ryzyko!

  • Nigdy nie używaj prawdziwych danych klientów w testach
  • Środowisko testowe powinno mieć anonimizowane dane
  • RODO/GDPR - dane osobowe podlegają ochronie prawnej
  • Używaj generowanych danych lub danych fikcyjnych

Repozytorium testów jest często dostępne dla wielu osób - dane klientów nie powinny tam trafić.

Bardzo ważne w kontekście Rossmanna - firma przetwarza dane milionów klientów. Jeśli dane produkcyjne trafią do repozytorium testowego - to naruszenie RODO. Zawsze używajcie danych fikcyjnych lub zanonimizowanych.
