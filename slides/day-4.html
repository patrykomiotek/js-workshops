Powtórzenie dnia 3

Quiz - Tablice

Co zwróci poniższy kod?


const fruits = ["apple", "banana", "cherry"];
const result = fruits.filter(f => f.length > 5);
console.log(result);
  1. A) ["apple", "banana", "cherry"]
  2. B) ["banana", "cherry"]
  3. C) ["banana"]
  4. D) ["cherry"]

Odpowiedź: B - "banana" (6 liter) i "cherry" (6 liter)

Rozgrzewka z dnia 3. Filter zwraca nową tablicę z elementami spełniającymi warunek. "apple" ma 5 liter - nie przechodzi warunku > 5.

Powtórzenie - Obiekty

Quiz - Destrukturyzacja

Co wyświetli console.log?


const user = { name: "Anna", age: 30, city: "Łódź" };
const { name, ...rest } = user;
console.log(rest);

Odpowiedź: { age: 30, city: "Łódź" }

Spread/rest operator w destrukturyzacji - zbiera pozostałe właściwości do nowego obiektu. To ważny wzorzec, często spotykany w kodzie React.

Powtórzenie - Metody tablicowe

Quiz - map vs forEach

Jaka jest różnica?


const numbers = [1, 2, 3];

// Wariant A
const doubled = numbers.map(n => n * 2);

// Wariant B
const result = [];
numbers.forEach(n => result.push(n * 2));

Kluczowa różnica

map() zwraca nową tablicę, forEach() zwraca undefined. Oba warianty dają ten sam efekt, ale map jest bardziej deklaratywny.

Przypomnienie kluczowej różnicy. W praktyce map jest preferowany gdy chcemy transformować dane, forEach gdy chcemy wykonać efekt uboczny.
01

DOM - Document Object Model

Praca z dokumentem HTML w JavaScript

Przechodzimy do pierwszego dużego bloku - DOM. To fundament pracy testera automatycznego. Zrozumienie DOM pozwoli lepiej pisać selektory w Cypress czy Playwright.

Czym jest DOM?

  • DOM = Document Object Model
  • Przeglądarka zamienia HTML na drzewo obiektów
  • JavaScript może to drzewo czytać i modyfikować
  • Każdy tag HTML = węzeł (node) w drzewie

<html>
  <body>
    <h1>Tytuł</h1>
    <p>Tekst</p>
  </body>
</html>

document → html → body → h1, p

DOM to interfejs programistyczny do dokumentów HTML. Przeglądarka parsuje HTML i tworzy drzewo obiektów, po którym możemy nawigować i które możemy modyfikować z poziomu JS.

Drzewo DOM - wizualizacja


<html>
  <head>
    <title>Moja strona</title>
  </head>
  <body>
    <div id="app">
      <h1 class="title">Witaj!</h1>
      <ul>
        <li>Punkt 1</li>
        <li>Punkt 2</li>
      </ul>
    </div>
  </body>
</html>

DevTools

Otwórz F12 → zakładka Elements, aby zobaczyć drzewo DOM na żywo.

Pokaz na żywo w DevTools - jak drzewo DOM odpowiada kodowi HTML. Warto pokazać, że zmiany w DOM są natychmiastowe i widoczne.

document.querySelector()

Znajduje pierwszy element pasujący do selektora CSS


// Szukanie po tagu
const heading = document.querySelector("h1");

// Szukanie po klasie
const title = document.querySelector(".title");

// Szukanie po ID
const app = document.querySelector("#app");

// Szukanie po atrybucie
const input = document.querySelector('[data-testid="email"]');

// Zagnieżdżony selektor
const firstItem = document.querySelector("ul > li:first-child");

Selektor CSS = Selektor Cypress

cy.get('.title') działa dokładnie tak samo jak querySelector('.title')

querySelector to najczęściej używana metoda. Przyjmuje dowolny selektor CSS. Zwraca pierwszy pasujący element lub null jeśli nie znajdzie. Podkreśl związek z Cypress.

document.querySelectorAll()

Znajduje wszystkie elementy pasujące do selektora


// Zwraca NodeList (podobny do tablicy)
const items = document.querySelectorAll("li");

console.log(items.length); // np. 5

// Iteracja po elementach
items.forEach(item => {
  console.log(item.textContent);
});

// Konwersja na tablicę (jeśli potrzebujemy map/filter)
const itemsArray = [...items];
const longItems = itemsArray.filter(i => i.textContent.length > 10);

Uwaga

NodeList to nie tablica - ma forEach, ale nie ma map ani filter. Użyj spread: [...nodeList]

querySelectorAll zwraca NodeList. Ważne - to nie jest pełna tablica. Ma forEach, ale żeby użyć map/filter, trzeba najpierw zamienić na tablicę operatorem spread.

document.getElementById()

Szybkie szukanie po atrybucie id


// HTML: <div id="login-form">...</div>

const form = document.getElementById("login-form");

// Równoważne z:
const formAlt = document.querySelector("#login-form");

getElementById

  • Szuka tylko po id
  • Nieco szybszy
  • Starsze API

querySelector

  • Dowolny selektor CSS
  • Bardziej elastyczny
  • Nowoczesne API
getElementById to starsze API, ale nadal popularne. W praktyce querySelector jest bardziej elastyczny i zalecany. Warto znać oba, bo spotkacie je w kodzie.

Manipulacja tekstem


const heading = document.querySelector("h1");

// Odczyt tekstu
console.log(heading.textContent); // "Witaj!"

// Zmiana tekstu
heading.textContent = "Nowy tytuł";

// innerHTML - pozwala wstawiać HTML
const container = document.querySelector("#app");
container.innerHTML = "<strong>Pogrubiony</strong> tekst";

Bezpieczeństwo

innerHTML może być niebezpieczny (XSS), jeśli wstawiamy dane od użytkownika. Preferuj textContent gdy nie potrzebujesz HTML.

textContent to bezpieczny sposób na manipulację tekstem. innerHTML pozwala wstawić HTML, ale uwaga na XSS - nigdy nie wstawiamy niezaufanych danych przez innerHTML. Jako testerzy powinniście to sprawdzać!

Manipulacja stylami


const box = document.querySelector(".box");

// Bezpośrednia zmiana stylu
box.style.backgroundColor = "red";
box.style.fontSize = "24px";
box.style.display = "none"; // ukrywanie elementu

// Lepsze podejście - klasy CSS
box.classList.add("active");       // dodaj klasę
box.classList.remove("active");    // usuń klasę
box.classList.toggle("active");    // przełącz (dodaj/usuń)
box.classList.contains("active");  // sprawdź: true/false

Dobra praktyka

Zamiast zmieniać style bezpośrednio, dodawaj/usuwaj klasy CSS. Łatwiej testować i utrzymywać.

Dwa podejścia do stylów. style.property zmienia inline, classList operuje na klasach CSS. classList jest preferowany, bo separuje logikę od prezentacji. Dla testerów ważne - klasy są dobrymi selektorami.

Atrybuty elementów


const link = document.querySelector("a");

// Odczyt atrybutu
const href = link.getAttribute("href");
console.log(href); // "https://example.com"

// Ustawienie atrybutu
link.setAttribute("target", "_blank");

// Sprawdzenie czy atrybut istnieje
link.hasAttribute("rel"); // true/false

// Usunięcie atrybutu
link.removeAttribute("target");

// data-* atrybuty (specjalne API)
const card = document.querySelector('[data-product-id="123"]');
console.log(card.dataset.productId); // "123"
Atrybuty HTML są dostępne przez get/set/has/removeAttribute. Specjalne data-* atrybuty mają wygodne API przez dataset. W testach data-testid jest bardzo popularny.

Tworzenie elementów


// 1. Utwórz element
const newItem = document.createElement("li");

// 2. Dodaj treść
newItem.textContent = "Nowy punkt";

// 3. Dodaj klasy
newItem.classList.add("item", "new");

// 4. Dodaj do dokumentu
const list = document.querySelector("ul");
list.appendChild(newItem);

// Wstawienie przed innym elementem
const firstItem = list.querySelector("li");
list.insertBefore(newItem, firstItem);

// Nowoczesna metoda: append (przyjmuje tekst i elementy)
list.append(newItem, " - dodano!");
Tworzenie elementów to trzy kroki: createElement, ustawienie właściwości, appendChild/append. insertBefore pozwala wstawić w konkretne miejsce. append to nowoczesna wersja appendChild.

Usuwanie elementów


// Nowoczesny sposób - metoda remove()
const item = document.querySelector(".old-item");
item.remove();

// Starszy sposób - przez rodzica
const parent = document.querySelector("ul");
const child = parent.querySelector("li:last-child");
parent.removeChild(child);

// Usunięcie wszystkich dzieci
const container = document.querySelector("#list");
container.innerHTML = ""; // szybki sposób

Podpowiedź

.remove() jest prostsze i bardziej czytelne. Starsze podejście z removeChild spotkacie w starszym kodzie.

Usuwanie elementów - dwa podejścia. remove() jest nowoczesne i proste. removeChild to starsze API, ale nadal spotykane. innerHTML = "" to szybki sposób na wyczyszczenie kontenera.

Nawigacja po DOM


const item = document.querySelector("li.active");

// Rodzic
item.parentElement;          // <ul>

// Dzieci
item.parentElement.children; // HTMLCollection [li, li, li]

// Rodzeństwo
item.nextElementSibling;     // następny <li>
item.previousElementSibling; // poprzedni <li>

// Pierwsze/ostatnie dziecko
const list = document.querySelector("ul");
list.firstElementChild;      // pierwszy <li>
list.lastElementChild;       // ostatni <li>

// Najbliższy przodek pasujący do selektora
item.closest("div.container"); // szuka w górę drzewa
Nawigacja po drzewie DOM pozwala poruszać się między elementami bez ponownego wyszukiwania. closest() jest szczególnie przydatne - szuka w górę drzewa. Często używane w delegacji zdarzeń.

DOM a testowanie

Dlaczego tester powinien rozumieć DOM?

Cypress


// Selektory CSS
cy.get('[data-testid="login"]');
cy.get('.btn-primary');
cy.get('#email');

// Nawigacja
cy.get('li').first();
cy.get('li').eq(2);
cy.get('.item').parent();

DOM (JavaScript)


// Te same selektory!
querySelector('[data-testid="login"]');
querySelector('.btn-primary');
querySelector('#email');

// Nawigacja
querySelector('li:first-child');
querySelectorAll('li')[2];
querySelector('.item').parentElement;

Kluczowy wniosek

Cypress używa tych samych selektorów CSS co DOM. Umiejętność pracy z DOM = lepsza praca z Cypress!

To kluczowy slajd motywacyjny. Selektory CSS z DOM to dokładnie te same selektory używane w Cypress. Zrozumienie DOM jest bezpośrednio przekładalne na lepsze testy.

Praktyczny przykład

Scenariusz: weryfikacja formularza logowania

Przykład bliski pracy testera - weryfikacja formularza logowania. Pokazujemy jak odczytywać wartości pól, sprawdzać stan disabled, czytać komunikaty błędów.

02

Eventy

Zdarzenia w przeglądarce

Przechodzimy do zdarzeń - to mechanizm, dzięki któremu strony reagują na akcje użytkownika. Kliknięcia, wpisywanie tekstu, wysyłanie formularzy - to wszystko eventy.

Czym jest event?

  • Event = zdarzenie, które zachodzi w przeglądarce
  • Użytkownik klika przycisk → event "click"
  • Użytkownik wpisuje tekst → event "input"
  • Strona się załadowała → event "load"
  • Formularz wysłany → event "submit"

Popularne typy eventów

TypKiedy?
clickKliknięcie elementu
inputZmiana wartości pola (na bieżąco)
changeZmiana wartości pola (po opuszczeniu)
submitWysłanie formularza
keydownNaciśnięcie klawisza
mouseoverNajechanie myszką
Przegląd najczęstszych eventów. Dla testerów kluczowe: click, input, change, submit. To te same zdarzenia, które Cypress symuluje w testach.

addEventListener()

Nasłuchiwanie na zdarzenia


const button = document.querySelector("#myButton");

// Dodaj nasłuchiwanie na kliknięcie
button.addEventListener("click", function() {
  console.log("Przycisk kliknięty!");
});

// Z arrow function
button.addEventListener("click", () => {
  console.log("Kliknięto!");
});

// Nazwana funkcja (łatwiej usunąć)
function handleClick() {
  console.log("Kliknięto!");
}

button.addEventListener("click", handleClick);
button.removeEventListener("click", handleClick);
addEventListener to podstawowy sposób dodawania obsługi zdarzeń. Pierwszy argument to typ zdarzenia (string), drugi to funkcja callback. Warto wspomnieć o removeEventListener - wymaga tej samej referencji do funkcji.

Obiekt Event

Każdy handler otrzymuje obiekt zdarzenia z informacjami


button.addEventListener("click", function(event) {
  // Na jakim elemencie kliknięto?
  console.log(event.target);       // <button id="myButton">

  // Jaki typ zdarzenia?
  console.log(event.type);         // "click"

  // Pozycja myszy
  console.log(event.clientX);      // 150
  console.log(event.clientY);      // 300

  // Modyfikatory klawiszy
  console.log(event.ctrlKey);      // true/false
  console.log(event.shiftKey);     // true/false
});

event.target

event.target to element, który faktycznie wywołał zdarzenie. Niezwykle przydatne przy delegacji zdarzeń.

Event object zawiera mnóstwo informacji. Najważniejsze: target (kto wywołał), type (jaki typ), oraz metody jak preventDefault i stopPropagation.

Click events - przykład


// Licznik kliknięć
let count = 0;
const counter = document.querySelector("#counter");
const button = document.querySelector("#increment");

button.addEventListener("click", () => {
  count++;
  counter.textContent = `Kliknięć: ${count}`;
});

// Toggle widoczności
const toggleBtn = document.querySelector("#toggle");
const panel = document.querySelector("#panel");

toggleBtn.addEventListener("click", () => {
  panel.classList.toggle("hidden");
  toggleBtn.textContent = panel.classList.contains("hidden")
    ? "Pokaż"
    : "Ukryj";
});
Dwa praktyczne przykłady: licznik kliknięć i toggle widoczności panelu. Oba wzorce są bardzo popularne w aplikacjach webowych.

Input events


const searchInput = document.querySelector("#search");
const results = document.querySelector("#results");

// "input" - reaguje na KAŻDĄ zmianę (w trakcie pisania)
searchInput.addEventListener("input", (event) => {
  const query = event.target.value;
  results.textContent = `Szukasz: "${query}"`;
});

// "change" - reaguje PO opuszczeniu pola
const selectBox = document.querySelector("#category");

selectBox.addEventListener("change", (event) => {
  console.log(`Wybrano: ${event.target.value}`);
});

input vs change

input - odpala się przy każdym znaku (na bieżąco).
change - odpala się po opuszczeniu pola lub wybraniu z listy.

Kluczowa różnica: input odpala się na bieżąco (każdy keystroke), change dopiero po blur. Dla pól select, change odpala się od razu po wyborze opcji.

Form events i preventDefault()


const form = document.querySelector("#loginForm");

form.addEventListener("submit", (event) => {
  // Zapobiega domyślnemu zachowaniu (wysłanie formularza + przeładowanie)
  event.preventDefault();

  // Pobierz dane z formularza
  const email = form.querySelector("#email").value;
  const password = form.querySelector("#password").value;

  // Walidacja
  if (!email.includes("@")) {
    alert("Nieprawidłowy email!");
    return;
  }

  console.log("Logowanie:", { email, password });
  // Tu normalnie: fetch() do API...
});

preventDefault()

Blokuje domyślne zachowanie przeglądarki. Bez niego formularz przeładowałby stronę!

preventDefault to jedna z najważniejszych metod event object. Przy formularzach jest niezbędne, bo domyślnie przeglądarka przeładowuje stronę. W testach Cypress robi to automatycznie.

Delegacja zdarzeń

Zamiast nasłuchiwać na każdym elemencie - nasłuchuj na rodzicu

Bez delegacji


// 100 przycisków = 100 listenerów!
document.querySelectorAll(".btn")
  .forEach(btn => {
    btn.addEventListener("click",
      () => {
        console.log("Klik!");
      });
  });

Z delegacją


// 1 listener na rodzicu!
document.querySelector("#list")
  .addEventListener("click",
    (event) => {
      if (event.target
          .matches(".btn")) {
        console.log("Klik!");
      }
    });

Dlaczego to ważne?

Delegacja działa nawet dla elementów dodanych dynamicznie. Jeden listener zamiast setek = lepsza wydajność.

Delegacja zdarzeń to zaawansowana technika, ale warto ją znać. Zdarzenia "bąbelkują" w górę drzewa DOM, więc możemy nasłuchiwać na rodzicu. Działa też dla elementów dodanych dynamicznie.

03

Callbacki

Funkcje jako argumenty

Teraz przechodzimy od DOM do asynchroniczności. Zaczynamy od callbacków - najprostszego mechanizmu obsługi operacji asynchronicznych w JS.

Funkcje jako argumenty

W JavaScript funkcje to wartości - można je przekazywać jako argumenty


// Callback = funkcja przekazana jako argument do innej funkcji
function greet(name, callback) {
  const message = `Cześć, ${name}!`;
  callback(message);
}

// Użycie
greet("Anna", function(msg) {
  console.log(msg); // "Cześć, Anna!"
});

// Z arrow function
greet("Jan", (msg) => console.log(msg));

// Już to znamy!
const numbers = [3, 1, 2];
numbers.sort((a, b) => a - b);  // (a, b) => a - b to callback
numbers.forEach(n => console.log(n)); // n => console.log(n) to callback
Callback to po prostu funkcja przekazana jako argument. Uczestnicy już je znają z metod tablicowych - forEach, map, filter - wszystkie używają callbacków.

setTimeout i setInterval


// setTimeout - wykonaj RAZ po opóźnieniu
console.log("Start");

setTimeout(() => {
  console.log("Minęły 2 sekundy!");
}, 2000); // czas w milisekundach

console.log("Koniec");
// Wypisze: "Start", "Koniec", (po 2s) "Minęły 2 sekundy!"

  seconds++;
  console.log(`Upłynęło: ${seconds}s`);

setTimeout i setInterval to podstawowe funkcje asynchroniczne. Kluczowe obserwacje: console.log("Koniec") wypisze się PRZED "Minęły 2 sekundy" - to jest asynchroniczność! clearInterval/clearTimeout pozwalają anulować odliczanie.

Asynchroniczność w akcji


console.log("1. Zaczynam");

setTimeout(() => {
  console.log("2. Timeout 0ms");
}, 0);

console.log("3. Kończę");
1. Zaczynam
3. Kończę
2. Timeout 0ms

Zaskoczenie!

Nawet z opóźnieniem 0ms, callback z setTimeout wykonuje się po bieżącym kodzie. JavaScript najpierw kończy to, co robi, potem sprawdza kolejkę zadań.

To kluczowy moment zrozumienia asynchroniczności. Nawet setTimeout z 0ms nie jest synchroniczny. JS ma event loop - najpierw wykonuje cały synchroniczny kod, potem sprawdza kolejkę callbacków.

Callback Hell

Co się stanie, gdy mamy wiele operacji zależnych od siebie?


// "Piramida zagłady" - callback hell
fetchUser(userId, (user) => {
  fetchOrders(user.id, (orders) => {
    fetchOrderDetails(orders[0].id, (details) => {
      fetchProduct(details.productId, (product) => {
        console.log(product.name);
        // ...i to może iść dalej
      }, (error) => { console.error(error); });
    }, (error) => { console.error(error); });
  }, (error) => { console.error(error); });
}, (error) => { console.error(error); });

Problemy

  • Kod jest trudny do czytania
  • Obsługa błędów jest powtarzalna
  • Trudno debugować

Rozwiązanie? Promise!

Callback hell to klasyczny problem - zagnieżdżone callbacki tworzą "piramidę zagłady". Kod staje się nieczytelny i trudny do utrzymania. To motywacja do nauki Promise.
04

Promise

Obietnica wyniku w przyszłości

Promise to mechanizm, który rozwiązuje problem callback hell. Pozwala pisać asynchroniczny kod w sposób bardziej liniowy i czytelny.

Czym jest Promise?

Promise = obietnica, że operacja zakończy się w przyszłości

Analogia z życia

  • Zamawiasz pizzę online
  • Dostajesz numer zamówienia (Promise)
  • Pizza jeszcze nie gotowa (pending)
  • Pizza dostarczona (fulfilled)
  • Albo: restauracja zamknięta (rejected)

Trzy stany

StanOpis
pendingOczekiwanie na wynik
fulfilledSukces - mamy dane
rejectedBłąd - coś poszło nie tak
Użyj analogii z pizzą - uczestnicy łatwiej zapamiętają. Promise to jak paragon z numerem zamówienia - obiecuje wynik w przyszłości. Trzy stany: pending, fulfilled, rejected.

Tworzenie Promise


const myPromise = new Promise((resolve, reject) => {
  // Symulacja operacji asynchronicznej
  const success = true;

  setTimeout(() => {
    if (success) {
      resolve("Dane pobrane!"); // Sukces → fulfilled
    } else {
      reject("Błąd połączenia"); // Błąd → rejected
    }
  }, 1000);
});

console.log(myPromise); // Promise { <pending> }

Kluczowe

  • resolve(wartość) - "obietnica spełniona, oto wynik"
  • reject(powód) - "obietnica odrzucona, oto przyczyna"
Konstruktor Promise przyjmuje funkcję executor z dwoma parametrami: resolve i reject. Wewnątrz wykonujemy operację asynchroniczną i wywołujemy odpowiednią funkcję. W praktyce rzadko tworzymy Promise ręcznie - częściej używamy funkcji, które je zwracają (np. fetch).

.then() - obsługa sukcesu


const myPromise = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve({ name: "Anna", age: 30 });
  }, 1000);
});

// .then() - co zrobić gdy Promise się spełni
myPromise.then((data) => {
  console.log("Sukces!");
  console.log(data.name); // "Anna"
});

// Bardziej życiowy przykład
function fetchUser(id) {
  return new Promise((resolve) => {
    setTimeout(() => {
      resolve({ id, name: "Jan", email: "jan@rossmann.pl" });
    }, 500);
  });
}

fetchUser(1).then(user => {
  console.log(`Witaj, ${user.name}!`);
});
.then() rejestruje funkcję, która wykona się gdy Promise przejdzie do stanu fulfilled. Argument callbacka to wartość przekazana do resolve().

.catch() - obsługa błędu


function fetchData(shouldFail) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (shouldFail) {
        reject(new Error("Serwer niedostępny!"));
      } else {
        resolve({ data: "OK" });
      }
    }, 500);
  });
}

// Obsługa sukcesu i błędu
fetchData(false)
  .then(result => {
    console.log("Dane:", result.data); // "Dane: OK"
  })
  .catch(error => {
    console.error("Błąd:", error.message);
  });

// Z błędem
fetchData(true)
  .then(result => console.log(result))   // pomijane!
  .catch(error => console.error(error.message)); // "Serwer niedostępny!"
.catch() łapie błędy z Promise (reject) oraz błędy rzucone wewnątrz .then(). Zawsze dodawaj .catch() - bez niego błędy mogą zostać "połknięte" bez informacji.

Chainowanie Promise

Każde .then() zwraca nowy Promise - można je łączyć w łańcuch


fetchUser(1)
  .then(user => {
    console.log(`Użytkownik: ${user.name}`);
    return fetchOrders(user.id); // zwraca nowy Promise
  })
  .then(orders => {
    console.log(`Zamówień: ${orders.length}`);
    return fetchOrderDetails(orders[0].id); // kolejny Promise
  })
  .then(details => {
    console.log(`Produkt: ${details.productName}`);
  })
  .catch(error => {
    // Łapie błąd z DOWOLNEGO kroku!
    console.error("Coś poszło nie tak:", error.message);
  });

Porównaj z callback hell

Ten sam kod, ale płaski zamiast zagnieżdżony. Jeden .catch() obsługuje błędy z całego łańcucha.

Chainowanie to kluczowa zaleta Promise nad callbackami. Kod jest liniowy, a nie zagnieżdżony. Jeden catch na końcu łapie błędy z dowolnego kroku - ogromne uproszczenie.

Promise.all()

Czekanie na wiele Promise jednocześnie


const promise1 = fetch("/api/users").then(r => r.json());
const promise2 = fetch("/api/products").then(r => r.json());
const promise3 = fetch("/api/orders").then(r => r.json());

// Czekaj na WSZYSTKIE
Promise.all([promise1, promise2, promise3])
  .then(([users, products, orders]) => {
    console.log("Użytkownicy:", users.length);
    console.log("Produkty:", products.length);
    console.log("Zamówienia:", orders.length);
  })
  .catch(error => {
    // Jeśli KTÓRYKOLWIEK się nie uda
    console.error("Błąd:", error.message);
  });

Uwaga

Jeśli jakikolwiek Promise zostanie odrzucony, cały Promise.all jest odrzucony. Użyj Promise.allSettled() jeśli chcesz wyniki niezależnie od błędów.

Promise.all uruchamia wiele Promise równolegle i czeka na wszystkie. Szybsze niż sekwencyjne wykonanie. Ale uwaga - fail-fast: jeden błąd odrzuca całość.

05

async/await

Asynchroniczny kod, który wygląda jak synchroniczny

async/await to składnia cukrowa nad Promise. Pozwala pisać asynchroniczny kod w sposób, który wygląda jak zwykły, synchroniczny kod.

async function


// Funkcja oznaczona "async" ZAWSZE zwraca Promise
async function greet(name) {
  return `Cześć, ${name}!`;
}

// To jest równoważne z:
function greetPromise(name) {
  return Promise.resolve(`Cześć, ${name}!`);
}

// Użycie - tak samo jak Promise
greet("Anna").then(msg => console.log(msg));
// "Cześć, Anna!"

// Arrow function też może być async
const fetchData = async () => {
  return { data: "OK" };
};

Kluczowa zasada

Słowo async przed funkcją sprawia, że automatycznie zwraca Promise.

async to modyfikator funkcji. Każda funkcja async automatycznie zwraca Promise. Wartość zwrócona przez return jest opakowywana w Promise.resolve().

await - czekanie na Promise


// await "rozpakowuje" Promise - czeka na wynik
async function loadUser() {
  console.log("Pobieram użytkownika...");

  const user = await getUser(1); // czeka na Promise
  console.log("Użytkownik:", user.name);

  const age = await getAge(user.name); // czeka na następny Promise
  console.log("Wiek:", age);

  return user;
}

loadUser();

Ważne

await można używać tylko wewnątrz funkcji async!

await pauzuje wykonanie funkcji async do momentu, gdy Promise się rozwiąże. Kod wygląda jak synchroniczny, ale jest asynchroniczny pod spodem. Kluczowe ograniczenie: await działa tylko wewnątrz async function.

Obsługa błędów: try/catch


async function loadUserSafe() {
  try {
    const user = await getUser(1);
    console.log("Użytkownik:", user.name);

    const age = await getAge(user.name);
    console.log("Wiek:", age);

  } catch (error) {
    // Łapie rejected Promise (jak .catch())
    console.error("Błąd:", error.message);

  } finally {
    // Wykonuje się ZAWSZE (opcjonalne)
    console.log("Zakończono pobieranie danych");
  }
}

loadUserSafe();

try/catch = .catch()

To odpowiednik .catch() z łańcucha Promise. Łapie błędy z dowolnego await w bloku try.

try/catch to naturalna obsługa błędów w async/await, analogiczna do .catch() w Promise. finally wykonuje się niezależnie od wyniku - przydatne np. do ukrycia loadera.

Porównanie stylów

Promise .then()

    .then(user => {
      console.log(user.name);

Co wybrać?

Oba podejścia są poprawne. async/await jest zazwyczaj czytelniejszy, szczególnie przy wielu sekwencyjnych operacjach. W nowym kodzie preferuj async/await.

Ten sam kod w dwóch stylach. async/await jest bardziej czytelny i łatwiejszy do zrozumienia. Warto znać oba, bo stary kod często używa .then(). Nowy kod - preferuj async/await.

Fetch API

Pobieranie danych z serwera

Ostatni duży blok dnia - Fetch API. Tu łączymy wszystko: DOM, eventy, async/await i Promise. Fetch to sposób, w jaki przeglądarka komunikuje się z serwerem.

Czym jest Fetch API?

  • Fetch = wbudowana funkcja do komunikacji z serwerem
  • Wysyła żądania HTTP (GET, POST, PUT, DELETE...)
  • Zwraca Promise z odpowiedzią serwera
  • Zastąpił starszego XMLHttpRequest

// Najprostsze użycie
fetch("https://jsonplaceholder.typicode.com/users/1")
  .then(response => response.json())
  .then(data => console.log(data));

Dla testerów

Fetch to ta sama technologia, którą aplikacja używa "pod spodem". Zrozumienie fetch = zrozumienie komunikacji klient-serwer.

Fetch to natywne API przeglądarki do żądań HTTP. Zwraca Promise, więc idealnie pasuje do tego, czego się właśnie nauczyliśmy. jsonplaceholder to darmowe API do testów.

GET request - pobieranie danych

Obsługa błędów w fetch


async function fetchSafe(url) {
  try {
    const response = await fetch(url);

    // UWAGA: fetch NIE rzuca błędu przy 404/500!
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    const data = await response.json();
    return data;

  } catch (error) {
    // Łapie: brak sieci, błąd parsowania JSON, nasz throw
    console.error("Błąd pobierania:", error.message);
    return null;
  }
}

// Użycie
const user = await fetchSafe("https://jsonplaceholder.typicode.com/users/999");
// HTTP 404 → "Błąd pobierania: HTTP 404: Not Found"

Pułapka fetch!

fetch nie rzuca błędu przy kodach 4xx/5xx. Rzuca błąd tylko przy problemach sieciowych. Musisz ręcznie sprawdzać response.ok!

To najczęstszy błąd z fetch - ludzie zakładają, że 404 rzuci wyjątek, ale NIE. fetch rzuca wyjątek tylko przy problemach sieciowych (brak połączenia, CORS). Zawsze sprawdzaj response.ok!

POST request - wysyłanie danych


async function createPost(title, body) {
  const response = await fetch("https://jsonplaceholder.typicode.com/posts", {
    method: "POST",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      title: title,
      body: body,
      userId: 1
    })
  });

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }

  const newPost = await response.json();
  console.log("Utworzono post:", newPost);
  return newPost;
}

// Użycie
await createPost("Mój tytuł", "Treść posta...");
// → { id: 101, title: "Mój tytuł", body: "Treść posta...", userId: 1 }
POST request wymaga obiektu opcji z method, headers i body. JSON.stringify zamienia obiekt JS na string JSON - serwer oczekuje stringa, nie obiektu. Content-Type mówi serwerowi, że wysyłamy JSON.
