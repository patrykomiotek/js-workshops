Powtórka z dnia 1

Quiz: Zmienne

Jaka jest różnica między let a const?

Odpowiedź: let pozwala na ponowne przypisanie wartości, const - nie. Oba mają zasięg blokowy.

Dajemy uczestnikom chwilę na zastanowienie się, potem odkrywamy odpowiedź. Przypominamy, że const nie oznacza niezmienności obiektów - tylko brak ponownego przypisania.

Powtórka z dnia 1

Quiz: Typy danych

Co zwróci typeof null?

Odpowiedź: "object" - to znany bug w JavaScript, który istnieje od początku języka.

Typowe pytanie na rozmowach kwalifikacyjnych. Warto znać tę ciekawostkę. typeof undefined zwraca "undefined", typeof NaN zwraca "number".

Powtórka z dnia 1

Quiz: Operatory

Co zwróci wyrażenie "5" == 5? A co "5" === 5?

Odpowiedź:
"5" == 5 → true (porównanie z konwersją typów)
"5" === 5 → false (ścisłe porównanie, różne typy)

Przypomnienie o ważności używania potrójnego znaku równości. W testach automatycznych zawsze używamy === żeby uniknąć niespodzianek.

02

Instrukcje warunkowe

Podejmowanie decyzji w kodzie

Przechodzimy do pierwszego nowego tematu. Instrukcje warunkowe to podstawa sterowania przepływem programu. Pozwalają na wykonywanie różnego kodu w zależności od warunków.

Instrukcja if

Podstawowa składnia - wykonaj kod jeśli warunek jest spełniony


let testsPassed = 15;
let totalTests = 20;

if (testsPassed === totalTests) {
  console.log("Wszystkie testy przeszly!");
}

Zasada

Warunek w nawiasach () jest konwertowany na true lub false.

Tłumaczenie składni if. Warunek musi być w nawiasach okrągłych, blok kodu w klamrowych. Jeśli warunek jest prawdziwy - kod się wykonuje.

Instrukcja if...else

Dwie ścieżki - co robić, gdy warunek nie jest spełniony


let statusCode = 404;

if (statusCode === 200) {
  console.log("Odpowiedz OK");
} else {
  console.log("Blad! Kod statusu: " + statusCode);
}
> Błąd! Kod statusu: 404
Dodajemy blok else, który wykonuje się gdy warunek jest fałszywy. Przykład z kontekstu testowania - sprawdzamy kod odpowiedzi HTTP.

Wiele warunków: else if


let statusCode = 301;

if (statusCode === 200) {
  console.log("Sukces");
} else if (statusCode === 301) {
  console.log("Przekierowanie");
} else if (statusCode === 404) {
  console.log("Nie znaleziono");
} else if (statusCode === 500) {
  console.log("Blad serwera");
} else {
  console.log("Nieznany kod: " + statusCode);
}
Możemy dodać dowolną liczbę bloków else if. Warunki są sprawdzane od góry do dołu - pierwszy spełniony warunek kończy sprawdzanie.

Zagnieżdżone warunki

Jeden if wewnątrz drugiego


let userRole = "admin";
let isLoggedIn = true;

if (isLoggedIn) {
  if (userRole === "admin") {
    console.log("Panel administratora");
  } else {
    console.log("Panel uzytkownika");
  }
} else {
  console.log("Prosze sie zalogowac");
}

Uwaga

Unikaj zbyt głębokich zagnieżdżeń - pogarszają czytelność kodu.

Pokaz zagnieżdżonego if. Wspomnienie o zasadzie - jeśli masz więcej niż 2-3 poziomy zagnieżdżenia, warto przemyśleć refaktoryzację. Można użyć operatorów logicznych && zamiast zagnieżdżania.

Operatory logiczne w warunkach

Zamiast zagnieżdżania - łączymy warunki


let isLoggedIn = true;
let userRole = "admin";

// Zamiast zagniezdzonego if:
if (isLoggedIn && userRole === "admin") {
  console.log("Panel administratora");
}

// Sprawdzanie kilku mozliwosci:
let browser = "Chrome";
if (browser === "Chrome" || browser === "Firefox") {
  console.log("Wspierana przegladarka");
}
Pokazujemy jak operatory logiczne && i || pomagają uniknąć zagnieżdżania. Częściej używamy tego zapisu w praktyce - jest czytelniejszy.

Operator trójargumentowy (ternary)

Skrócony zapis if...else w jednej linii


let testsPassed = 18;
let totalTests = 20;

// Pelny if/else:
let status;
if (testsPassed === totalTests) {
  status = "PASS";
} else {
  status = "FAIL";
}

// Ternary - to samo w jednej linii:
let status2 = (testsPassed === totalTests) ? "PASS" : "FAIL";

console.log(status2); // "FAIL"

Składnia

warunek ? wartość_jeśli_true : wartość_jeśli_false

Ternary jest przydatny dla prostych warunków. Nie należy go zagnieżdżać - wtedy staje się nieczytelny. Idealny do przypisywania wartości.

Instrukcja switch

Alternatywa dla wielu else if


let testStatus = "failed";

switch (testStatus) {
  case "passed":
    console.log("Test przeszedl pomyslnie");
    break;
  case "failed":
    console.log("Test nie przeszedl");
    break;
  case "skipped":
    console.log("Test zostal pominiety");
    break;
  default:
    console.log("Nieznany status: " + testStatus);
}
Switch porównuje wartość wyrażenia z każdym case za pomocą ===. Ważne: każdy case wymaga break! Bez break kod "przechodzi" dalej do następnego case (fall-through).

switch - pamiętaj o break!

Bez break kod wykonuje się dalej (fall-through)

Bez break (błąd!):


let priority = "high";
switch (priority) {
  case "high":
    console.log("Wysoki");
  case "medium":
    console.log("Sredni");
  case "low":
    console.log("Niski");
}
// Wyswietli: Wysoki, Sredni, Niski

Z break (poprawnie):


let priority = "high";
switch (priority) {
  case "high":
    console.log("Wysoki");
    break;
  case "medium":
    console.log("Sredni");
    break;
  case "low":
    console.log("Niski");
    break;
}
// Wyswietli: Wysoki
Pokaz fall-through - jeden z najczęstszych błędów. Można też celowo go wykorzystać, np. grupując case'y bez kodu między nimi.

Kiedy switch, kiedy if/else?

Cecha switch if/else
Porównanie Jedna wartość z wieloma opcjami Różne warunki, zakresy
Czytelność Lepsza przy wielu wariantach Lepsza przy 2-3 warunkach
Zakresy Nie obsługuje (np. x > 5) Obsługuje dowolne wyrażenia
Typy Używane z === Dowolne porównania

Wskazówka

Używaj switch gdy porównujesz jedną zmienną z wieloma konkretnymi wartościami.

Reguła kciuka: jeśli sprawdzamy jedną zmienną z wieloma możliwymi wartościami - switch. Jeśli mamy różne warunki lub zakresy - if/else.

Przykład: Walidacja formularza


let username = "jan_kowalski";
let password = "abc123";

if (username.length === 0) {
  console.log("Nazwa uzytkownika jest wymagana");
} else if (username.length < 3) {
  console.log("Nazwa musi miec min. 3 znaki");
} else if (password.length < 6) {
  console.log("Haslo musi miec min. 6 znakow");
} else {
  console.log("Formularz poprawny - mozna wyslac");
}
> Formularz poprawny - można wysłać
Przykład z kontekstu testowania - walidacja danych formularza. To typowy scenariusz, który testerzy spotykają codziennie.

Przykład: Sprawdzanie roli użytkownika


let userRole = "editor";

switch (userRole) {
  case "admin":
    console.log("Dostep: pelny");
    console.log("Moze usuwac uzytkownikow");
    break;
  case "editor":
    console.log("Dostep: edycja tresci");
    console.log("Nie moze usuwac uzytkownikow");
    break;
  case "viewer":
    console.log("Dostep: tylko odczyt");
    break;
  default:
    console.log("Nieznana rola - brak dostepu");
}
Praktyczny przykład ze switch - sprawdzamy rolę użytkownika i na jej podstawie wyświetlamy uprawnienia. To częsty scenariusz w testach aplikacji webowych.

Wartości truthy i falsy

JavaScript automatycznie konwertuje wartości na true/false

Falsy (fałszywe):


false
0
"" // pusty string
null
undefined
NaN

Truthy (prawdziwe):


true
42         // kazda liczba != 0
"tekst"    // niepusty string
[]         // tablica (nawet pusta!)
{}         // obiekt (nawet pusty!)
            

let errorMessage = "";
if (errorMessage) {
  console.log("Jest blad: " + errorMessage);
} else {
  console.log("Brak bledow"); // To sie wykona
}
Ważne w kontekście warunków - wiele wartości jest automatycznie konwertowanych. Pusta tablica i pusty obiekt są truthy! To częsty błąd początkujących.

Czas na praktykę!

Ćwiczenie: Klasyfikator ocen

  1. Zadeklaruj zmienną score z wynikiem testu (0-100)
  2. Napisz warunki, które wyświetlą ocenę:
    • 90-100: "Celujący"
    • 75-89: "Dobry"
    • 60-74: "Dostateczny"
    • Poniżej 60: "Niedostateczny"
  3. Dodaj obsługę błędnych wartości (ujemne, większe niż 100)
Ćwiczenie na 10-15 minut. Przechodzimy po sali i pomagamy. Ważne, żeby uczestnicy spróbowali sami - nie podpowiadamy od razu.
03

Pętle

Powtarzanie operacji wiele razy

Przechodzimy do pętli. Pętle pozwalają na powtarzanie bloku kodu wielokrotnie. To kluczowe np. przy przetwarzaniu list wyników testów.

Pętla for

Najczęściej używana pętla - znamy liczbę powtórzeń


//    inicjalizacja; warunek; krok
for (let i = 0; i < 5; i++) {
  console.log("Iteracja numer: " + i);
}
> Iteracja numer: 0
> Iteracja numer: 1
> Iteracja numer: 2
> Iteracja numer: 3
> Iteracja numer: 4

Trzy części pętli for

Inicjalizacja - wykonuje się raz | Warunek - sprawdzany przed każdą iteracją | Krok - po każdej iteracji

Tłumaczenie każdej części pętli for. Zwrócenie uwagi na i++ (inkrementacja). Zmienna i jest dostępna tylko wewnątrz pętli dzięki let.

for - iterowanie po tablicy


let browsers = ["Chrome", "Firefox", "Safari", "Edge"];

for (let i = 0; i < browsers.length; i++) {
  console.log("Test w przegladarce: " + browsers[i]);
}
> Test w przeglądarce: Chrome
> Test w przeglądarce: Firefox
> Test w przeglądarce: Safari
> Test w przeglądarce: Edge

Wskazówka

browsers.length zwraca liczbę elementów tablicy. Indeksy zaczynają się od 0.

Klasyczne użycie pętli for - przechodzenie po elementach tablicy. Zwrócenie uwagi na .length i na to, że indeksy zaczynają się od 0.

Pętla while

Powtarzaj dopóki warunek jest spełniony


let attempts = 0;
let maxAttempts = 3;

while (attempts < maxAttempts) {
  console.log("Proba polaczenia nr " + (attempts + 1));
  attempts++;
}

console.log("Wykonano " + attempts + " prob");
> Próba połączenia nr 1
> Próba połączenia nr 2
> Próba połączenia nr 3
> Wykonano 3 prób
While używamy, gdy nie znamy dokładnej liczby powtórzeń. Warunek jest sprawdzany PRZED każdą iteracją - jeśli na początku jest fałszywy, kod się nie wykona ani razu.

Nieskończona pętla

Uwaga!

Jeśli warunek while nigdy nie stanie się false, pętla nigdy się nie zakończy!


// NIEBEZPIECZNE - nie uruchamiaj!
let counter = 0;
while (counter < 10) {
  console.log(counter);
  // Brak counter++ - petla nigdy sie nie skonczy!
}

Jak uniknąć?

Zawsze upewnij się, że warunek pętli w końcu stanie się false. W przeglądarce można zamknąć kartę. W Node.js: Ctrl+C.

Ważne ostrzeżenie! Pokazujemy przykład nieskończonej pętli i jak ją przerwać. To częsty błąd początkujących - zapominają o inkrementacji licznika.

Pętla do...while

Wykona się co najmniej raz - warunek sprawdzany na końcu


let userInput;
let validInputs = ["tak", "nie"];

do {
  userInput = "tak"; // symulacja odpowiedzi uzytkownika
  console.log("Uzytkownik wpisal: " + userInput);
} while (!validInputs.includes(userInput));

console.log("Zaakceptowano odpowiedz: " + userInput);

while: sprawdza warunek przed iteracją

do...while: sprawdza warunek po iteracji

do...while gwarantuje przynajmniej jedno wykonanie. Używana rzadziej, ale przydatna np. przy walidacji danych - chcemy przynajmniej raz poprosić o dane i potem sprawdzić czy są poprawne.

Pętla for...of

Wygodna składnia do iterowania po kolekcjach


let testResults = ["passed", "failed", "passed", "skipped", "passed"];

for (let result of testResults) {
  console.log("Wynik testu: " + result);
}
> Wynik testu: passed
> Wynik testu: failed
> Wynik testu: passed
> Wynik testu: skipped
> Wynik testu: passed

Zaleta

Nie potrzebujemy indeksu i - bezpośredni dostęp do elementu. Czytelniejszy kod!

for...of to nowoczesna składnia (ES6). Idealna, gdy nie potrzebujemy indeksu, a jedynie wartości elementów. Działa z tablicami, stringami i innymi obiektami iterowalnymi.

break - przerywanie pętli

Natychmiast kończy wykonywanie pętli


let testResults = ["passed", "passed", "failed", "passed", "passed"];

console.log("Szukam pierwszego bledu...");

for (let i = 0; i < testResults.length; i++) {
  if (testResults[i] === "failed") {
    console.log("Znaleziono blad w tescie nr " + (i + 1));
    break; // Konczymy szukanie
  }
  console.log("Test " + (i + 1) + ": OK");
}
> Szukam pierwszego błędu...
> Test 1: OK
> Test 2: OK
> Znaleziono błąd w teście nr 3
Break przerywa całą pętlę. Przydatne, gdy szukamy pierwszego wystąpienia i nie musimy przeglądać reszty kolekcji.

continue - pomijanie iteracji

Przeskakuje do następnej iteracji pętli


let testResults = ["passed", "skipped", "failed", "passed", "skipped"];

console.log("Analiza wynikow (pomijam skipped):");

for (let result of testResults) {
  if (result === "skipped") {
    continue; // Pomijamy pominiete testy
  }
  console.log("Analizuje wynik: " + result);
}
> Analiza wyników (pomijam skipped):
> Analizuję wynik: passed
> Analizuję wynik: failed
> Analizuję wynik: passed
Continue pomija bieżącą iterację i przechodzi do następnej. Nie kończy pętli jak break - tylko pomija resztę kodu w tej iteracji.

Zagnieżdżone pętle

Pętla wewnątrz pętli - np. testowanie wielu kombinacji


let browsers = ["Chrome", "Firefox"];
let resolutions = ["1920x1080", "1366x768", "375x667"];

for (let browser of browsers) {
  for (let resolution of resolutions) {
    console.log(`Test: ${browser} @ ${resolution}`);
  }
}
> Test: Chrome @ 1920x1080
> Test: Chrome @ 1366x768
> Test: Chrome @ 375x667
> Test: Firefox @ 1920x1080
> Test: Firefox @ 1366x768
> Test: Firefox @ 375x667
Zagnieżdżone pętle generują iloczyn kartezjański - każda kombinacja. Przykład z kontekstu testowania - sprawdzamy każdą przeglądarkę w każdej rozdzielczości. Uwaga na wydajność przy dużych zbiorach!

Przykład: Zliczanie wyników testów


let results = ["passed", "failed", "passed", "passed",
               "skipped", "failed", "passed"];

let passed = 0;
let failed = 0;
let skipped = 0;

for (let result of results) {
  if (result === "passed") passed++;
  else if (result === "failed") failed++;
  else if (result === "skipped") skipped++;
}

console.log(`Wyniki: ${passed} OK, ${failed} bledow, ${skipped} pominieto`);
console.log(`Procent sukcesu: ${Math.round(passed / results.length * 100)}%`);
> Wyniki: 4 OK, 2 błędów, 1 pominięto
> Procent sukcesu: 57%
Łączenie pętli z warunkami - typowy scenariusz. Zliczamy wyniki testów i obliczamy procent sukcesu. Uczestnicy mogą spróbować dodać więcej statystyk.

Przykład: Wyszukiwanie błędu


let logs = [
  "INFO: Aplikacja uruchomiona",
  "INFO: Uzytkownik zalogowany",
  "WARNING: Wolne polaczenie",
  "ERROR: Baza danych niedostepna",
  "INFO: Ponowna proba polaczenia"
];

let errorFound = false;

for (let log of logs) {
  if (log.startsWith("ERROR")) {
    console.log("Znaleziono blad: " + log);
    errorFound = true;
    break;
  }
}

if (!errorFound) {
  console.log("Brak bledow w logach");
}
Praktyczny przykład - przeszukiwanie logów w poszukiwaniu błędów. Używamy startsWith() do sprawdzenia początku stringa i break po znalezieniu.

Czas na praktykę!

Ćwiczenie: Pętle

  1. Stwórz tablicę z 5 nazwami produktów (np. z rossmann.pl)
  2. Użyj pętli for, aby wyświetlić każdy produkt z numerem: "1. Szampon", "2. Krem"...
  3. Użyj pętli for...of, aby policzyć ile produktów ma nazwę dłuższą niż 5 znaków
  4. Użyj pętli while, aby wyświetlać produkty aż napotkasz produkt o nazwie "STOP" (dodaj "STOP" do tablicy)
Ćwiczenie na 15 minut. Trzy podpunkty - każdy używa innej pętli. Pomagamy uczestnikom, zwracamy uwagę na poprawne warunki i inkrementację.

Ćwiczenie dodatkowe

Ćwiczenie: Raport testów

  1. Dana jest tablica wyników: ["pass", "fail", "pass", "pass", "fail", "skip", "pass"]
  2. Użyj pętli, aby policzyć ile jest wyników każdego typu
  3. Wyświetl raport: "Passed: 4 | Failed: 2 | Skipped: 1"
  4. Wyświetl procentowy udział każdego typu
Ćwiczenie dla szybszych uczestników. Łączy pętle z warunkami. Można zaproponować użycie switch zamiast if/else.
04

Funkcje

Organizacja i ponowne użycie kodu

Najważniejszy blok dnia! Funkcje to fundament programowania. Pozwalają na organizację kodu i jego ponowne użycie.

Czym jest funkcja?

Funkcja to nazwany blok kodu, który można wielokrotnie wywołać

Przepis kucharski

  • Ma nazwę ("Naleśniki")
  • Przyjmuje składniki (parametry)
  • Wykonuje kroki (ciało funkcji)
  • Daje wynik (return)

Funkcja w JS


function makePancakes(eggs, milk) {
  let batter = mix(eggs, milk);
  let result = fry(batter);
  return result;
}
Analogia z przepisem kucharskim jest bardzo intuicyjna. Funkcja ma nazwę, parametry (składniki), ciało (instrukcje) i może zwracać wynik (gotowe danie).

Deklaracja funkcji


// Deklaracja (definicja) funkcji
function greetTester(name) {
  console.log("Czesc, " + name + "! Gotowy na testy?");
}

// Wywolanie funkcji
greetTester("Anna");
greetTester("Marek");
> Cześć, Anna! Gotowy na testy?
> Cześć, Marek! Gotowy na testy?

Składnia

function nazwaFunkcji(parametry) { ... }

Pierwsza forma deklaracji - słowo kluczowe function, nazwa, nawiasy z parametrami, klamry z ciałem. Zwracamy uwagę na różnicę między deklaracją a wywołaniem.

Deklaracja vs wywołanie

Deklaracja tworzy funkcję. Wywołanie uruchamia ją.


// 1. Deklaracja - tworzymy funkcje (nic sie nie dzieje)
function runTest(testName) {
  console.log("Uruchamiam test: " + testName);
  console.log("Test zakonczony!");
}

// 2. Bez wywolania - funkcja istnieje, ale sie nie wykonuje
// ...nic w konsoli...

// 3. Wywolanie - uruchamiamy funkcje (teraz sie wykonuje!)
runTest("Login");
runTest("Wyszukiwanie");

Częsty błąd

Napisanie funkcji, ale zapomnienie o jej wywołaniu. Funkcja sama się nie uruchomi!

Podkreślamy różnicę - deklaracja to jak napisanie przepisu, wywołanie to jak gotowanie według tego przepisu. Można wywołać wielokrotnie.

Parametry i argumenty


//                parametr1    parametr2
function calculateSum(a, b) {
  console.log("Suma: " + (a + b));
}

//           argument1  argument2
calculateSum(10, 20);     // Suma: 30
calculateSum(5, 3);       // Suma: 8

Różnica

Parametr - zmienna w deklaracji funkcji (placeholder).
Argument - konkretna wartość przekazana przy wywołaniu.

Parametr to slot, argument to wartość, która ten slot wypełnia. W praktyce ludzie używają tych terminów zamiennie, ale warto znać różnicę.

return - zwracanie wartości

Funkcja może zwrócić wynik, który można zapisać w zmiennej


function add(a, b) {
  return a + b;
  // Kod po return NIE zostanie wykonany!
}

let result = add(15, 25);
console.log("Wynik: " + result); // Wynik: 40

// Mozna tez uzyc bezposrednio:
console.log("Podwojony: " + add(10, 10) * 2); // Podwojony: 40

Ważne

return kończy wykonywanie funkcji. Kod po return nigdy się nie wykona.

Return zwraca wartość i kończy funkcję. Bez return funkcja zwraca undefined. Pokaz, że console.log WYŚWIETLA, a return ZWRACA - to różne rzeczy.

console.log vs return

console.log - wyświetla


function addAndPrint(a, b) {
  console.log(a + b);
}

let result = addAndPrint(2, 3);
// Wyswietli: 5
console.log(result);
// Wyswietli: undefined

return - zwraca


function addAndReturn(a, b) {
  return a + b;
}

let result = addAndReturn(2, 3);
// Nic nie wyswietli
console.log(result);
// Wyswietli: 5

Zasada

console.log = pokazuje na ekranie. return = oddaje wartość do dalszego użycia.

To jeden z najczęstszych błędów początkujących - mylenie console.log z return. Console.log jest jak powiedzenie czegoś na głos, return to jak podanie wyniku.

Parametry domyślne

Można podać wartość domyślną, gdy argument nie zostanie przekazany


function createTestReport(testName, status = "pending") {
  return `Test "${testName}" - status: ${status}`;
}

console.log(createTestReport("Login", "passed"));
// Test "Login" - status: passed

console.log(createTestReport("Wyszukiwanie"));
// Test "Wyszukiwanie" - status: pending

Składnia

function nazwa(param1, param2 = "domyślna") { ... }

Parametry domyślne są bardzo przydatne. Jeśli nie przekażemy argumentu, użyty zostanie domyślny. Stosowane często w funkcjach konfiguracyjnych.

Wyrażenie funkcyjne (Function Expression)

Funkcja przypisana do zmiennej


// Function Expression
const calculatePercentage = function(part, total) {
  return Math.round((part / total) * 100);
};

console.log(calculatePercentage(15, 20) + "%"); // 75%

Function Declaration:


// Mozna wywolac PRZED deklaracja
sayHi(); // Dziala!
function sayHi() {
  console.log("Czesc!");
}

Function Expression:


// NIE mozna wywolac przed
sayHi(); // Blad!
const sayHi = function() {
  console.log("Czesc!");
};
Różnica: deklaracja jest "hoistowana" - można ją wywołać przed definicją. Expression nie - musi być zdefiniowana przed wywołaniem. W praktyce stosujemy obie formy.

Arrow Function (funkcja strzałkowa)

Krótsza składnia - wprowadzona w ES6


// Function Expression
const add = function(a, b) {
  return a + b;
};

// Arrow Function - to samo, krocej
const addArrow = (a, b) => {
  return a + b;
};

console.log(add(2, 3));      // 5
console.log(addArrow(2, 3)); // 5

Składnia

Zamieniamy function na => (strzałkę) po nawiasach.

Arrow function to nowoczesna składnia, bardzo popularna w dzisiejszym JS. Pokazujemy transformację krok po kroku: usuwamy function, dodajemy strzałkę.

Arrow Function - skróty


// Pelna arrow function
const multiply = (a, b) => {
  return a * b;
};

// Skrocona - jedna linia, implicit return (bez return i {})
const multiplyShort = (a, b) => a * b;

// Jeden parametr - mozna pominac nawiasy
const double = x => x * 2;

// Brak parametrow - puste nawiasy
const getTimestamp = () => Date.now();

console.log(multiplyShort(3, 4)); // 12
console.log(double(5));           // 10
Trzy skróty: 1) implicit return - bez klamr i return dla jednego wyrażenia, 2) jeden parametr - bez nawiasów, 3) brak parametrów - puste nawiasy wymagane.

Porównanie: 3 sposoby tworzenia funkcji

Cecha Declaration Expression Arrow
Składnia function f() {} const f = function() {} const f = () => {}
Hoisting Tak Nie Nie
Krótki zapis Nie Nie Tak (implicit return)
Użycie Ogólne Callbacki Callbacki, metody tablic

Dla początkujących

Na start używaj function declaration - jest najprostsza. Arrow functions poznasz lepiej przy tablicach.

Podsumowanie trzech sposobów. Nie przytłaczamy szczegółami - na początku ważne jest, żeby uczestnicy umieli tworzyć proste funkcje. Arrow functions będą bardziej użyteczne przy tablicach (metody map, filter).

Scope - zasięg zmiennych

Gdzie zmienna jest dostępna (widoczna)?


let globalVar = "Jestem globalna";  // Dostepna wszedzie

function testScope() {
  let localVar = "Jestem lokalna"; // Tylko wewnatrz funkcji
  console.log(globalVar);  // OK
  console.log(localVar);   // OK
}

testScope();
console.log(globalVar);  // OK
console.log(localVar);   // BLAD! localVar nie istnieje tutaj

Global scope - dostępna wszędzie

Local scope - dostępna tylko w funkcji

Scope to zasięg widoczności zmiennej. Zmienna lokalna nie jest widoczna na zewnątrz funkcji. Globalna - jest widoczna wszędzie. Dobra praktyka: unikaj zmiennych globalnych.

Block scope - let vs var


// var - NIE ma zasiegu blokowego (tylko funkcyjny)
if (true) {
  var oldWay = "Widoczna na zewnatrz!";
}
console.log(oldWay); // "Widoczna na zewnatrz!" - dziala!

// let - MA zasieg blokowy
if (true) {
  let newWay = "Tylko wewnatrz bloku";
}
console.log(newWay); // BLAD! newWay nie istnieje tutaj

Dlaczego unikamy var?

var "wyciekała" z bloków if, for itd. let i const są bezpieczniejsze - mają zasięg blokowy.

Ważna różnica - var ma tylko zasięg funkcyjny, więc "wycieka" z bloków. To był częsty źródło błędów. let i const rozwiązują ten problem. Dlatego w nowoczesnym JS używamy let i const, nie var.

Scope w pętlach


// Problem z var w petli
for (var i = 0; i < 3; i++) {
  // ...
}
console.log(i); // 3 - var "wycieklo" z petli!

// Rozwiazanie z let
for (let j = 0; j < 3; j++) {
  // ...
}
console.log(j); // BLAD! j nie istnieje tutaj

Zasada

Zawsze używaj let lub const. Unikaj var - to relikt przeszłości.

Praktyczny przykład - zmienna i z pętli for jest dostępna po pętli gdy użyjemy var. Z let - nie. To ważne przy pisaniu testów, gdzie zmienne pętli nie powinny wpływać na inne części kodu.

Funkcje jako "czarne skrzynki"

Input → Przetwarzanie → Output


// Nie musimy wiedziec JAK dziala wewnatrz
// Wazne: co przyjmuje i co zwraca

function validateEmail(email) {
  // Wewnetrzna logika (czarna skrzynka)
  let hasAt = email.includes("@");
  let hasDot = email.includes(".");
  let isLongEnough = email.length >= 5;

  return hasAt && hasDot && isLongEnough;
}

// Uzycie - interesuje nas tylko input/output
console.log(validateEmail("jan@rossmann.pl"));  // true
console.log(validateEmail("niepoprawny"));       // false
Analogia z czarną skrzynką - używający funkcji nie musi znać implementacji, wystarczy wiedzieć co wchodzi i co wychodzi. To kluczowa koncepcja w programowaniu i w testowaniu!

Przykład: Formatowanie daty testu


function formatTestDate(date) {
  let day = String(date.getDate()).padStart(2, "0");
  let month = String(date.getMonth() + 1).padStart(2, "0");
  let year = date.getFullYear();
  let hours = String(date.getHours()).padStart(2, "0");
  let minutes = String(date.getMinutes()).padStart(2, "0");

  return `${day}.${month}.${year} ${hours}:${minutes}`;
}

let now = new Date();
console.log("Test wykonany: " + formatTestDate(now));
// np. "Test wykonany: 27.02.2026 14:30"
Praktyczny przykład - formatowanie daty. Taka funkcja jest używana wielokrotnie w raportach testowych. Pokazujemy padStart i getMonth()+1.

Przykład: Walidacja hasła


const validatePassword = (password) => {
  if (password.length < 8) {
    return "Haslo za krotkie (min. 8 znakow)";
  }
  if (password === password.toLowerCase()) {
    return "Haslo musi zawierac wielka litere";
  }
  if (!/\d/.test(password)) {
    return "Haslo musi zawierac cyfre";
  }
  return "Haslo poprawne";
};

console.log(validatePassword("abc"));       // Za krotkie
console.log(validatePassword("abcdefgh"));  // Brak wielkiej litery
console.log(validatePassword("Abcdefgh"));  // Brak cyfry
console.log(validatePassword("Abcdefg1"));  // Poprawne
Arrow function z wieloma warunkami walidacji. Funkcja zwraca komunikat błędu lub potwierdzenie. To typowy scenariusz testowy - sprawdzanie reguł walidacji.

Funkcje wywołujące funkcje

Funkcje mogą korzystać z innych funkcji


function isValidEmail(email) {
  return email.includes("@") && email.includes(".");
}

function isValidPassword(password) {
  return password.length >= 8;
}

function validateForm(email, password) {
  if (!isValidEmail(email)) {
    return "Nieprawidlowy email";
  }
  if (!isValidPassword(password)) {
    return "Haslo za krotkie";
  }
  return "Formularz poprawny";
}

console.log(validateForm("jan@test.pl", "haslo123"));
// "Formularz poprawny"
Kompozycja funkcji - duża funkcja korzysta z mniejszych. To dobra praktyka - każda funkcja robi jedną rzecz. Łatwiejsze do testowania i debugowania.
