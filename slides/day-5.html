Testy E2E - Warsztat Praktyczny

Szkolenie JavaScript dla testerów manualnych

Rossmann - Bottega

Ostatni dzień szkolenia - dzień praktyczny! Dziś uczestnicy zastosują wszystko, czego nauczyli się w ciągu ostatnich 4 dni. Główna część dnia to warsztat, w którym wspólnie zbudujemy testy E2E dla aplikacji rossmann-test-app. Zachęć uczestników do aktywnego kodowania.

Agenda

  • Powtórzenie dnia 4 ~30 min
  • Testowanie dynamicznych danych ~1h
  • Dobre praktyki programistyczne w testach ~1h
  • Warsztat - Budujemy testy dla rossmann-test-app ~3h
  • Podsumowanie szkolenia ~30 min
Omów plan dnia. Podkreśl, że dzisiejszy dzień jest w dużej mierze praktyczny. Po krótkim powtórzeniu i dwóch blokach teorii z przykładami, przejdziemy do 3-godzinnego warsztatu. Przerwy będą elastyczne - dostosowane do tempa grupy.

Powtórzenie: Strategie selektorów

Uszereguj selektory od najlepszego do najgorszego w kontekście testów:

  1. page.locator('.btn-primary')
  2. page.getByTestId('submit-btn')
  3. page.locator('#app > div:nth-child(3) > button')
  4. page.getByRole('button', { name: 'Wyślij' })

Odpowiedź: 2, 4, 1, 3 - data-testid jest najbardziej stabilny, zagnieżdżone selektory CSS są najgorsze

Przypomnij piramidę selektorów z dnia 4. data-testid to dedykowany atrybut testowy, nie zmienia się przy refaktoryzacji UI. Tekst przycisku jest dobry, ale może się zmienić przy tłumaczeniach. Klasa CSS może się zmienić przy redesignie. Zagnieżdżone selektory są kruche - każda zmiana w strukturze HTML je psuje.

Powtórzenie: Page Object Model

Co jest główną zaletą wzorca Page Object Model?

  1. Testy działają szybciej
  2. Nie trzeba pisać selektorów
  3. Zmiana UI wymaga aktualizacji tylko w jednym miejscu
  4. Testy automatycznie się naprawiają

Odpowiedź: 3 - POM enkapsuluje selektory i akcje w jednym miejscu. Gdy zmieni się UI, aktualizujemy tylko Page Object.

POM to wzorzec projektowy, nie magiczne rozwiązanie. Wymaga pracy z góry, ale oszczędza ogromną ilość czasu przy utrzymaniu testów. Każda strona/komponent ma swoją klasę z selektorami i metodami akcji.

Powtórzenie: Co automatyzować?

Sklep internetowy Rossmann - który scenariusz jest NAJLEPSZYM kandydatem do automatyzacji?

  1. Sprawdzenie, czy nowy baner wygląda ładnie
  2. Proces zakupowy: dodanie do koszyka - checkout - potwierdzenie
  3. Jednorazowa migracja danych z innego systemu
  4. Testowanie animacji po najechaniu na produkt

Odpowiedź: 2 - Krytyczny, powtarzalny, stabilny scenariusz biznesowy - idealny kandydat do automatyzacji

Przypomnij kryteria: wysoka wartość biznesowa, powtarzalność, stabilność, możliwość weryfikacji. Baner i animacje to testy wizualne (inne narzędzia). Jednorazowa migracja nie wymaga automatyzacji. Proces zakupowy to rdzeń biznesu - musi działać zawsze.
01

Testowanie dynamicznych danych

Jak pisać stabilne testy, gdy dane się zmieniają?

To jeden z największych wyzwań w testach E2E. Dane dynamiczne to: daty, losowe ID, spersonalizowane treści, ceny po promocjach, rekomendacje. Nauczymy się strategii radzenia sobie z tymi problemami.

Problem: dane się zmieniają

Typowe źródła niestabilnych testów:

  • Daty i godziny - "Dziś jest 07.03.2026"
  • Losowe identyfikatory - "ORD-48291"
  • Spersonalizowane treści - rekomendacje, oferty
  • Ceny i promocje - zmieniają się w czasie
  • Dane z zewnętrznych API - mogą być niedostępne

Uwaga

Test, który przechodzi dzisiaj, a nie przechodzi jutro - to nie jest test, to loteria.

Poproś uczestników o przykłady z ich codziennej pracy - jakie dane się zmieniają w aplikacjach, które testują? To pomoże im powiązać teorię z praktyką.

Daty i czasy - kruche asercje

Źle - kruche


// Nie rób tak!
await expect(page.getByTestId('date'))
  .toHaveText('07.03.2026');

await expect(greeting).toHaveText(
  'Witaj, Anna! Dzis jest 07.03.2026'
);

Dobrze - elastyczne


// Regex na format daty
const dateText = await page
  .getByTestId('date').textContent();
expect(dateText).toMatch(/\d{2}\.\d{2}\.\d{4}/);

// Partial match - sprawdzaj stałą część
await expect(greeting)
  .toContainText('Witaj, Anna!');
Pokaż na żywo: uruchom test z hardkodowaną datą - przejdzie dziś, jutro nie. Strategia: sprawdzaj format daty, nie konkretną wartość. Albo sprawdzaj tylko część tekstu, która jest stała.

Strategia: zakresy dat i czasów


// Sprawdź timestamp w rozsądnym zakresie
test('should have recent order time', async ({ page }) => {
  const timestamp = await page.getByTestId('order-time')
    .getAttribute('data-timestamp');
  const orderTime = new Date(Number(timestamp));
  const now = new Date();
  const diffMinutes = (now - orderTime) / 1000 / 60;
  expect(diffMinutes).toBeLessThan(5);
});
    

// Sprawdź, że data jest "dzisiejsza"
test('should display today date', async ({ page }) => {
  await page.goto('/dashboard');

  const dateText = await page.getByTestId('current-date').textContent();
  const today = new Date();
  const expectedDay = String(today.getDate()).padStart(2, '0');
  const expectedMonth = String(today.getMonth() + 1).padStart(2, '0');

  expect(dateText).toContain(`${expectedDay}.${expectedMonth}`);
});
Dwa podejścia: generuj oczekiwaną datę dynamicznie albo sprawdzaj zakres. Unikaj hardkodowania dat - to najczęstsza przyczyna "zepsutych" testów w poniedziałek rano.

Losowe identyfikatory


// Sprawdzaj format, nie wartość
const orderId = await page.getByTestId('order-id').textContent();
expect(orderId).toMatch(/^ORD-\d+$/);

Regex w testach

/^ORD-\d+$/ - zaczyna się od "ORD-", potem jedna lub więcej cyfr. Sprawdzamy format, nie wartość.


// Inne przydatne wzorce regex:
/^[A-Z]{2}-\d{6}$/       // np. PL-123456
/^\w+@\w+\.\w+$/         // prosty email
/^\d{2}-\d{3}$/           // kod pocztowy
/^[0-9a-f]{8}-[0-9a-f]{4}/ // UUID (początek)
Wyjaśnij regex krok po kroku: ^ - początek, ORD- - literalny tekst, \d+ - cyfry, $ - koniec. Nie trzeba znać całego regex - wystarczy kilka podstawowych wzorców. Pokaż regex101.com jako narzędzie do testowania wyrażeń regularnych.

Spersonalizowane treści

Rekomendacje, oferty, propozycje - różne dla każdego użytkownika


// Źle - konkretny produkt może zniknąć z rekomendacji
await expect(page.getByTestId('recommendations'))
  .toContainText('Szampon XYZ');

// Dobrze - sprawdzaj strukturę, nie treść
await expect(page.getByTestId('recommendations')).toBeVisible();
await expect(page.getByTestId('product-card')).not.toHaveCount(0);

// Dobrze - sprawdzaj, że pierwsza karta ma wymagane elementy
const firstCard = page.getByTestId('product-card').first();
await expect(firstCard.getByTestId('product-name')).not.toBeEmpty();
await expect(firstCard.getByTestId('product-price')).not.toBeEmpty();
await expect(firstCard.getByTestId('add-to-cart')).toBeVisible();
Klucz: testuj STRUKTURĘ, nie TREŚĆ dynamiczną. Sprawdź, że sekcja rekomendacji istnieje, ma elementy, każdy element ma nazwę, cenę i przycisk. Nie sprawdzaj, JAKI konkretnie produkt jest wyświetlany.

Ceny i formatowanie walutowe


// Sprawdzaj format ceny, nie konkretną wartość
const priceFormat = await page
  .getByTestId('product-price').textContent();
expect(priceFormat).toMatch(/^\d+,\d{2}\s?zl$/);

// Sprawdzaj, że cena jest liczbą > 0
const priceText = await page.getByTestId('product-price').textContent();
const price = parseFloat(
  priceText.replace(',', '.').replace(/[^\d.]/g, '')
);
expect(price).toBeGreaterThan(0);

Kiedy sprawdzać konkretną cenę?

Tylko gdy kontrolujesz dane testowe - np. przez mocki API lub fixtures.

Ceny mogą się zmieniać w wyniku promocji, przeliczania walut itp. Lepiej sprawdzać format i zakres wartości niż konkretną cenę. Wyjątek: gdy mockujemy API - wtedy znamy dane.

Mockowanie API - kontrola nad danymi

Zamiast testować z prawdziwym API, przechwytujemy żądania i zwracamy kontrolowane dane

Zalety

  • Kontrolowane dane testowe
  • Testy nie zależą od backendu
  • Szybsze wykonanie
  • Można testować edge cases

Wady

  • Nie testujemy prawdziwego API
  • Mock może się rozjechać z realnym API
  • Wymaga utrzymania mocków
Wyjaśnij analogię: mock to jak dublerka w filmie. Wygląda jak prawdziwy aktor, ale kontrolujemy co robi. Testy z mockami są stabilne, ale nie zastąpią testów integracyjnych. Idealna strategia: większość testów z mockami + kilka smoke testów z prawdziwym API.

Mockowanie API - page.route()


import { test, expect } from '@playwright/test';

test('displays mocked products', async ({ page }) => {
  // Przechwytujemy żądanie do API
  await page.route('**/api/products', route => {
    route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([
        { id: 1, name: 'Test Product', price: 29.99 },
        { id: 2, name: 'Another Product', price: 49.99 }
      ])
    });
  });

  await page.goto('/products');

  // Teraz wiemy dokładnie, jakie dane zobaczymy
  await expect(page.getByTestId('product-card')).toHaveCount(2);
  await expect(page.getByText('Test Product')).toBeVisible();
  await expect(page.getByText('29,99 zl')).toBeVisible();
});
page.route() przechwytuje żądania HTTP pasujące do wzorca URL. Używamy ** jako wildcard dla dowolnego hosta. route.fulfill() zwraca kontrolowaną odpowiedź. Dzięki temu test nie zależy od stanu bazy danych ani dostępności backendu.

Kiedy mockować, kiedy nie?

Mockuj gdy:

  • Dane są dynamiczne/losowe
  • Backend jest niestabilny
  • Testujesz edge cases (błędy API)
  • Chcesz szybkie testy w CI
  • Zależy Ci na izolacji frontendowej

Nie mockuj gdy:

  • Smoke testy produkcyjne
  • Testy integracyjne API
  • Chcesz przetestować cały flow
  • Sprawdzasz wydajność

Złota zasada

80% testów z mockami (szybkie, stabilne) + 20% bez mocków (realistyczne).

To nie jest albo-albo. Najlepsza strategia to mieszanka. Większość testów E2E z mockami, a kilka kluczowych scenariuszy (happy path) z prawdziwym backendem. Smoke testy na produkcji powinny zawsze używać prawdziwego API.

Mockowanie błędów API


// Symulacja błędu 500
await page.route('**/api/products', route => {
  route.fulfill({
    status: 500,
    contentType: 'application/json',
    body: JSON.stringify({ error: 'Server Error' })
  });
});

await page.goto('/products');
await expect(page.getByTestId('error-message'))
  .toContainText('Wystąpił błąd');
    

// Playwright - symulacja błędu serwera
await page.route('**/api/products', route => {
  route.fulfill({ status: 500, body: 'Internal Server Error' });
});
await page.goto('/products');
await expect(page.getByText('Wystąpił błąd')).toBeVisible();

Testuj obsługę błędów

Możesz symulować różne kody: 400, 401, 403, 404, 500 - a nawet timeout sieciowy.

Testowanie obsługi błędów jest często pomijane, a to kluczowe dla UX. Czy aplikacja wyświetla przyjazny komunikat? Czy można spróbować ponownie? Mocki pozwalają łatwo symulować różne kody błędów.

Dobre praktyki w testach

Jak pisać testy, które łatwo utrzymać i zrozumieć

Ten blok to zbiór najważniejszych praktyk. Nie trzeba stosować wszystkich od razu - najważniejsze to zacząć od dobrej struktury i nazewnictwa, a resztę dodawać stopniowo.

DRY w testach - z umiarem

Źle - kopiuj/wklej


// Ten sam login w każdym teście
test('test 1', async ({ page }) => {
  await page.goto('/login');
  await page.getByTestId('email')
    .fill('test@rossmann.pl');
  await page.getByTestId('password')
    .fill('Test1234');
  await page.getByTestId('login-btn')
    .click();
  // ...reszta testu
});

test('test 2', async ({ page }) => {
  // Ten sam login skopiowany...
});

Ale nie przesadzaj!

Czytelność > DRY w testach


// Za dużo abstrakcji
test('test 1', async ({ page }) => {
  await doEverything(page, 'scenario-1');
});

// Lepiej - czytelne kroki
test('test 1', async ({ page }) => {
  await login(page, 'test@rossmann.pl');
  await addToCart(page, 'Szampon');
  await checkout(page);
  await verifyOrder(page);
});
Test powinien być czytelny jak historia. Każdy powinien szybko zrozumieć CO test robi, nawet bez czytania implementacji helperów. Za dużo abstrakcji sprawia, że trzeba przekopywać się przez wiele plików, żeby zrozumieć prosty test.

Helper functions - Playwright


// helpers/auth.ts
import { Page } from '@playwright/test';

export async function login(page: Page, email: string, password: string) {
  await page.goto('/login');
  await page.getByTestId('email').fill(email);
  await page.getByTestId('password').fill(password);
  await page.getByTestId('login-btn').click();
  await page.waitForURL('**/dashboard');
}

export async function addToCart(page: Page, productName: string) {
  await page.getByTestId('product-card')
    .filter({ hasText: productName })
    .getByTestId('add-to-cart').click();
}
    

// Użycie w teście - czyste i czytelne
test('should complete purchase flow', async ({ page }) => {
  await login(page, 'test@rossmann.pl', 'Test1234');
  await addToCart(page, 'Szampon');
  await expect(page.getByTestId('cart-badge')).toContainText('1');
});
W Playwright tworzymy zwykłe funkcje async, które przyjmują page jako argument. Importujemy je w testach jak zwykłe moduły. Alternatywnie można użyć fixtures (dependency injection) - pokażemy to na następnym slajdzie.

W Playwright: Fixtures

Odpowiednik Custom Commands z poprzedniego slajdu - w Playwright mechanizm dependency injection.


// fixtures.ts
import { test as base } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';
import { DashboardPage } from './pages/DashboardPage';

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  dashboardPage: async ({ page }, use) => {
    await use(new DashboardPage(page));
  },
});

export { expect } from '@playwright/test';
    

// Użycie w teście
import { test, expect } from './fixtures';

test('should login and see dashboard', async ({ loginPage, dashboardPage }) => {
  await loginPage.login('test@rossmann.pl', 'Test1234');
  await expect(dashboardPage.welcomeMessage).toBeVisible();
});
Fixtures w Playwright to mechanizm dependency injection. Deklarujemy zależności testu, a Playwright automatycznie je tworzy i niszczy. To mocniejszy mechanizm niż beforeEach - fixtures mogą mieć zależności między sobą i automatyczne czyszczenie.

Setup i teardown - hooki


import { test, expect } from '@playwright/test';
import { login } from './helpers/auth';

test.describe('Dashboard', () => {
  test.beforeEach(async ({ page }) => {
    // Wykonuje się PRZED KAŻDYM testem
    await login(page, 'test@rossmann.pl', 'Test1234');
  });

  test.afterEach(async ({ context }) => {
    // Wykonuje się PO KAŻDYM teście
    await context.clearCookies();
  });

  test('should display welcome message', async ({ page }) => {
    await expect(page.getByTestId('welcome')).toBeVisible();
  });

  test('should show product list', async ({ page }) => {
    const cards = page.getByTestId('product-card');
    await expect(cards).not.toHaveCount(0);
  });
});
beforeEach to idealne miejsce na powtarzalne kroki setup - logowanie, nawigacja, ustawienie stanu. afterEach to czyszczenie - ciasteczka, dane testowe. W Playwright każdy test domyślnie dostaje świeży kontekst przeglądarki, więc izolacja jest wbudowana.

Izolacja testów

Każdy test MUSI być niezależny

Testy nie mogą zależeć od kolejności wykonania ani od stanu zostawionego przez inny test.

Źle


// Test 1 tworzy produkt
test('creates product', async ({ page }) => {
  await createProduct(page, 'Szampon');
});

// Test 2 ZAKLADA, ze produkt istnieje
test('edits product', async ({ page }) => {
  await page.getByTestId('Szampon')
    .click(); // Nie zadziala solo!
});

Dobrze


// Każdy test tworzy własne dane
test('edits product', async ({ page }) => {
  // Arrange
  await createProduct(page, 'Szampon');

  // Act
  await page.getByTestId('Szampon').click();
  await page.getByTestId('name')
    .fill('Nowy Szampon');
  await page.getByTestId('save').click();

  // Assert
  await expect(page.getByText('Nowy Szampon'))
    .toBeVisible();
});
Testy mogą być uruchamiane w dowolnej kolejności, równolegle, lub pojedynczo. Jeśli test 2 wymaga danych z testu 1, to jest błąd projektowy. Każdy test powinien sam tworzyć potrzebne dane (lub używać mocków) i sprzątać po sobie.

Wzorzec Arrange-Act-Assert


test('should add product to cart and update total', async ({ page }) => {
  // ARRANGE - przygotuj warunki
  await page.route('**/api/products', route => {
    route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([{ id: 1, name: 'Szampon', price: 15.99 }])
    });
  });
  await page.goto('/products');

  // ACT - wykonaj akcje
  await page.getByTestId('add-to-cart').first().click();

  // ASSERT - sprawdz wynik
  await expect(page.getByTestId('cart-badge')).toContainText('1');
  await expect(page.getByTestId('cart-total')).toContainText('15,99');
});

AAA - czytelna struktura

Arrange (Przygotuj) - Act (Wykonaj) - Assert (Sprawdź). Każdy test powinien mieć te 3 fazy.

AAA to standard w testowaniu. Pomaga utrzymać testy czytelne i dobrze zorganizowane. Arrange: dane, mocki, nawigacja. Act: kliknięcie, wpisanie tekstu, wysłanie formularza. Assert: sprawdzenie wyniku. Nie mieszaj tych faz!

Dobra nazwa testu to dokumentacja

Czytając nazwy testów powinno się rozumieć, co aplikacja robi - bez czytania kodu.

Dobre nazwy testów to jak dokumentacja wymagań. Gdy test się wysypie w CI, z nazwy powinno być jasne, co nie działa. "test 1", "login test" to złe nazwy. "should redirect to dashboard when credentials are valid" - od razu wiadomo, co się stało.

Dane testowe w plikach JSON


// fixtures/users.json
{
  "validUser": {
    "email": "test@rossmann.pl",
    "password": "Test1234",
    "name": "Anna Testowa"
  },
  "invalidUser": {
    "email": "bad@rossmann.pl",
    "password": "wrong"
  }
}
    

// Playwright - import JSON jako moduł
import users from './fixtures/users.json';

test('should login with valid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.login(users.validUser.email, users.validUser.password);
  await expect(page.getByText(users.validUser.name)).toBeVisible();
});
Dane testowe w osobnych plikach JSON. Łatwo zmieniać, łatwo przeglądać, czytelne w kodzie. W Playwright importujemy JSON jak zwykły moduł (wymaga resolveJsonModule: true w tsconfig). Nie hardkoduj danych w testach - wyciągnij do fixtures.

CI/CD - testy w pipeline


# .github/workflows/e2e-tests.yml - Playwright
name: E2E Tests
on: [push, pull_request]

jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: playwright-report/

Automatyzacja testów

Testy E2E powinny uruchamiać się automatycznie przy każdym pushu i pull requeście.

GitHub Actions to jedno z wielu narzędzi CI - jest też GitLab CI, Jenkins, Azure DevOps. Kluczowe: testy uruchamiają się automatycznie, wyniki są widoczne w PR, raporty zapisywane jako artefakty. upload-artifact zachowuje raport HTML nawet gdy testy failują (if: failure()). npx playwright install --with-deps instaluje przeglądarki z zależnościami systemowymi.

Anty-wzorce w testach E2E

Unikaj tych błędów

  • Hardkodowane waity - await page.waitForTimeout(5000) zamiast czekania na element
  • Zależne testy - test B wymaga danych z testu A
  • Testowanie implementacji - sprawdzanie klas CSS zamiast zachowania
  • Za dużo w jednym teście - test sprawdza 20 rzeczy na raz
  • Brak cleanup - dane testowe zostają w bazie
  • Ignorowanie flaky testów - "czasem nie przechodzi, to skip"
Omów każdy anty-wzorzec z przykładem. Hardkodowane waity to najczęstszy problem - spowalniają testy i nadal mogą failować na wolnym CI. Flaky testy to tykająca bomba - ignorowanie ich prowadzi do utraty zaufania w cały suite testowy.

// tests/pages/LoginPage.ts
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitBtn: Locator;
  readonly error: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByTestId('login-email-input');
    this.passwordInput = page.getByTestId('login-password-input');
    this.submitBtn = page.getByTestId('login-submit-btn');
    this.error = page.getByTestId('login-error');
  }

  async visit() { await this.page.goto('/'); }

  async login(email: string, password: string) {
    await this.visit();
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
  }
}
W Playwright Page Object to klasa, która przyjmuje page w konstruktorze. Lokatory definiujemy jako pola klasy - są leniwe, nie wyszukują elementu od razu. Metoda login() łączy visit + wypełnienie formularza + submit. error to locator - można na nim wywołać asercje.

Toast: sprawdzamy, że pojawia się i ma odpowiedni tekst. Playwright automatycznie retryuje asercje, więc nie trzeba czekać na animację. Używamy Page Objects do logowania i akcji.

Rozwiązanie - testy zaawansowane (2/2)


// Test responsywnosci - mobilny viewport
test.describe('Mobile tests', () => {
  test.use({ viewport: { width: 390, height: 844 } });

  test('should adapt to mobile viewport', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.login('test@rossmann.pl', 'Test1234');

    // Menu mobilne powinno być widoczne
    await expect(page.getByTestId('mobile-menu')).toBeVisible();
    // Nawigacja desktopowa powinna być ukryta
    await expect(page.getByTestId('desktop-nav')).not.toBeVisible();
  });
});

// Alternatywa - użyj wbudowanych urządzeń Playwright
import { devices } from '@playwright/test';
test.use({ ...devices['iPhone 13'] });
Dwa podejścia do testowania responsywności. test.use() ustawia viewport dla grupy testów. Playwright ma wbudowane profile urządzeń (devices) z realistycznymi ustawieniami - viewport, userAgent, isMobile, hasTouch. Dostępne: iPhone 13, Pixel 5, Galaxy S9 itp.

Typowe problemy i rozwiązania

FAQ z warsztatu

  • "Test timeout" - element nie istnieje lub ma inny data-testid. Sprawdź w DevTools.
  • "Element not visible" - element jest w DOM, ale ukryty. Sprawdź CSS display/visibility.
  • "strict mode violation" - selektor pasuje do wielu elementów. Użyj .first(), .nth(0) lub bardziej precyzyjnego selektora.
  • "Timeout exceeded" - Playwright nie może znaleźć elementu w czasie timeout (domyślnie 30s).

Debugowanie

Użyj npx playwright test --ui do interaktywnego debugowania. Trace viewer pokazuje każdy krok z podglądem DOM.

Przejdź przez typowe problemy, które uczestnicy prawdopodobnie napotkali. Pokaż jak debugować każdy z nich. Playwright UI mode jest bardzo pomocny - pokazuje każdy krok testu z podglądem aplikacji i snapshotem DOM. Można też użyć await page.pause() do zatrzymania testu.
