Klasy w JavaScript

Szablony do tworzenia obiektów

Klasy to jeden z fundamentów programowania obiektowego. W kontekście testowania - klasy są podstawą wzorca Page Object Model.

Czym jest klasa?

Analogia

Klasa to szablon (przepis), a obiekt to instancja (gotowe danie).

  • Klasa User - szablon użytkownika
  • Obiekt jan - konkretny użytkownik
  • Można tworzyć wiele instancji z jednej klasy

// Klasa = szablon
class User {
  // ...definicja
}

// Instancja = konkretny obiekt
const jan = new User();
const anna = new User();
Porównaj klasę do formularza - formularz to szablon, wypełniony formularz to instancja. Do tej pory tworzyli obiekty literałami {}, teraz mamy szablon wielokrotnego użytku.

Składnia class


class Product {
  constructor(name, price) {
    this.name = name;
    this.price = price;
  }

  displayInfo() {
    return `${this.name} - ${this.price} zł`;
  }
}

const shampoo = new Product("Szampon", 12.99);
console.log(shampoo.displayInfo());
// "Szampon - 12.99 zł"

Konwencja

Nazwy klas zaczynamy wielką literą (PascalCase): Product, User, TestPage.

Omów każdą część: słowo kluczowe class, nazwa, ciało klasy w nawiasach klamrowych. Zwróć uwagę na PascalCase - to standard w JS.

constructor() - inicjalizacja


class User {
  constructor(name, email, role) {
    this.name = name;       // przypisanie do instancji
    this.email = email;
    this.role = role;
    this.createdAt = new Date(); // wartość domyślna
  }
}

const admin = new User("Jan", "jan@rossmann.pl", "admin");
console.log(admin.name);      // "Jan"
console.log(admin.createdAt); // aktualna data
  • constructor wywoływany automatycznie przy new
  • Przyjmuje argumenty i inicjalizuje właściwości
  • Każda klasa może mieć tylko jeden constructor
constructor to specjalna metoda - uruchamiana automatycznie przez operator new. Służy do ustawiania początkowych wartości obiektu. Pokaż, że createdAt jest ustawiana bez parametru - wartość domyślna.

Metody klasy


class Calculator {
  constructor() {
    this.result = 0;
  }

  add(value) {
    this.result += value;
    return this; // umożliwia łańcuchowanie
  }

  subtract(value) {
    this.result -= value;
    return this;
  }

  getResult() {
    return this.result;
  }
}

const calc = new Calculator();
calc.add(10).add(5).subtract(3);
console.log(calc.getResult()); // 12
Metody definiujemy bez słowa function - po prostu nazwa i nawiasy. Zwracanie this umożliwia łańcuchowanie metod (method chaining). Ten wzorzec jest używany np. w Cypress: cy.get().click().type().

this - kontekst obiektu


class User {
  constructor(name) {
    this.name = name;
  }

  greet() {
    return `Cześć, jestem ${this.name}`;
  }
}

const jan = new User("Jan");
console.log(jan.greet()); // "Cześć, jestem Jan"

const anna = new User("Anna");
console.log(anna.greet()); // "Cześć, jestem Anna"

Zapamiętaj

this odnosi się do bieżącej instancji - obiektu, na którym wywołujemy metodę.

this to referencja do aktualnego obiektu. Każda instancja ma swoje własne dane. Porównaj: jan.greet() - this wskazuje na jan, anna.greet() - this wskazuje na anna.

Tworzenie instancji: new


class TestCase {
  constructor(title, priority) {
    this.title = title;
    this.priority = priority;
    this.status = "pending";
  }

  run() {
    this.status = "running";
    console.log(`Uruchamiam: ${this.title}`);
  }
}

// Tworzenie instancji
const test1 = new TestCase("Login test", "high");
const test2 = new TestCase("Search test", "medium");

test1.run(); // "Uruchamiam: Login test"
console.log(test1.status); // "running"
console.log(test2.status); // "pending"
Operator new tworzy nowy obiekt, wywołuje constructor i zwraca obiekt. Każda instancja jest niezależna - zmiana w test1 nie wpływa na test2.

Gettery i settery


class Product {
  constructor(name, netPrice) {
    this.name = name;
    this._netPrice = netPrice; // konwencja: _ = prywatne
  }

  get price() {
    return this._netPrice * 1.23; // z VAT
  }

  set price(grossPrice) {
    if (grossPrice < 0) {
      throw new Error("Cena nie może być ujemna");
    }
    this._netPrice = grossPrice / 1.23;
  }
}

const item = new Product("Krem", 10);
console.log(item.price);   // 12.3 (get - jak właściwość)
item.price = 24.60;        // set - jak przypisanie
console.log(item._netPrice); // 20
Gettery i settery wyglądają jak zwykłe właściwości, ale pod spodem wykonują logikę. Konwencja _ (underscore) oznacza, że pole jest "prywatne" - nie powinno się go używać bezpośrednio.

Dziedziczenie: extends


class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} wydaje dźwięk`;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name} szczeka: Hau!`;
  }
}

class Cat extends Animal {
  speak() {
    return `${this.name} miauczy: Miau!`;
  }
}

const dog = new Dog("Burek");
console.log(dog.speak()); // "Burek szczeka: Hau!"
extends tworzy relację rodzic-dziecko. Klasa potomna dziedziczy metody i właściwości. Metody można nadpisywać (override) - jak speak() w Dog.

super() - wywołanie rodzica


class Person {
  constructor(name, age) {
    this.name = name;
    this.age = age;
  }

  introduce() {
    return `Jestem ${this.name}, mam ${this.age} lat`;
  }
}

class Employee extends Person {
  constructor(name, age, position) {
    super(name, age); // wywołanie constructora rodzica
    this.position = position;
  }

  introduce() {
    return `${super.introduce()}, pracuję jako ${this.position}`;
  }
}

const emp = new Employee("Jan", 30, "tester");
console.log(emp.introduce());
// "Jestem Jan, mam 30 lat, pracuję jako tester"
super() w constructorze musi być wywołane PRZED użyciem this. super.metoda() pozwala wywołać wersję metody z klasy rodzica.

Metody statyczne: static


class MathHelper {
  static add(a, b) {
    return a + b;
  }

  static isEven(number) {
    return number % 2 === 0;
  }

  static generateId() {
    return `ID-${Date.now()}`;
  }
}

// Wywołanie BEZ tworzenia instancji
console.log(MathHelper.add(2, 3));      // 5
console.log(MathHelper.isEven(4));      // true
console.log(MathHelper.generateId());   // "ID-1709..."

Kiedy static?

Gdy metoda nie potrzebuje danych instancji - narzędziowe funkcje, fabryki, helpery.

Metody statyczne należą do klasy, nie do instancji. Nie mają dostępu do this. Przykłady: Math.random(), Math.max(), Array.isArray().

Klasy w testowaniu - POM

Wzorzec Page Object Model - każda strona to osobna klasa:


class LoginPage {
  constructor() {
    this.usernameInput = '#username';
    this.passwordInput = '#password';
    this.submitButton = '#login-btn';
  }

  visit() {
    cy.visit('/login');
  }

  login(username, password) {
    cy.get(this.usernameInput).type(username);
    cy.get(this.passwordInput).type(password);
    cy.get(this.submitButton).click();
  }
}

// W teście:
const loginPage = new LoginPage();
loginPage.visit();
loginPage.login("admin", "haslo123");
Page Object Model to najważniejszy wzorzec w automatyzacji testów UI. Klasa enkapsuluje selektory i akcje dla jednej strony - łatwiejsze utrzymanie testów. Gdy zmieni się selektor - zmieniamy w jednym miejscu, nie w dziesiątkach testów.

Praktyczne przykłady klas


class Product {
  constructor(name, price, category) {
    this.name = name;
    this.price = price;
    this.category = category;
  }

  applyDiscount(percent) {
    this.price = this.price * (1 - percent / 100);
    return this;
  }

  toString() {
    return `${this.name} (${this.category}) - ${this.price.toFixed(2)} zł`;
  }
}

const products = [
  new Product("Szampon", 15.99, "Włosy"),
  new Product("Krem", 29.99, "Pielęgnacja"),
  new Product("Pasta", 8.49, "Higiena")

products.forEach(p => console.log(p.toString()));
Pokaż, jak klasy łączą się z tym, czego nauczyli się wcześniej - tablice, forEach, template literals. toString() jest automatycznie wywoływana przy konwersji do stringa.

Rozwiązanie: ShoppingCart


class ShoppingCart {
  constructor() {
    this.items = [];
  }

  addItem(product, quantity = 1) {
    this.items.push({ product, quantity });
  }

  getTotal() {
    return this.items.reduce(
      (sum, item) => sum + item.product.price * item.quantity, 0
    );
  }

  getItemCount() {
    return this.items.reduce((count, item) => count + item.quantity, 0);
  }

  removeItem(productName) {
    this.items = this.items.filter(
      item => item.product.name !== productName
    );
  }
}
Omów rozwiązanie krok po kroku. Zwróć uwagę na reduce - łączy się z dniem 3. Pokaż, jak filter działa przy usuwaniu.

Moduły ES

Organizacja kodu w plikach

Moduły pozwalają dzielić kod na mniejsze, łatwiejsze do zarządzania pliki. Każdy plik to osobny moduł ze swoim zakresem zmiennych.

Czym są moduły?

Bez modułów

  • Cały kod w jednym pliku
  • Konflikty nazw
  • Trudne utrzymanie
  • Brak enkapsulacji

Z modułami

  • Każdy plik = osobny moduł
  • Izolowane zakresy zmiennych
  • Jawne importy/eksporty
  • Łatwe ponowne użycie
Bez modułów: wyobraź sobie projekt z 50 plikami JS, gdzie każda zmienna globalna może kolidować z inną. Moduły rozwiązują ten problem - eksportujesz tylko to, co chcesz udostępnić.

export - eksportowanie

Named export


// utils.js
export function add(a, b) {
  return a + b;
}

export const PI = 3.14159;

export class Calculator {
  // ...
}

Default export


// User.js
export default class User {
  constructor(name) {
    this.name = name;
  }
}

// Tylko JEDEN default
// export na plik

Różnica

Named - wiele eksportów z pliku. Default - jeden główny eksport.

Named export - importujemy po nazwie w nawiasach klamrowych. Default export - importujemy bez nawiasów, pod dowolną nazwą. W praktyce: klasy często jako default, funkcje pomocnicze jako named.

import - importowanie


// Named imports - w nawiasach klamrowych
import { add, PI } from './utils.js';

// Default import - bez nawiasów
import User from './User.js';

// Alias - zmiana nazwy
import { add as sum } from './utils.js';

// Importowanie wszystkiego
import * as Utils from './utils.js';
console.log(Utils.add(2, 3)); // 5
console.log(Utils.PI);        // 3.14159

Ścieżka

Zawsze podawaj ścieżkę: './plik.js' lub '../folder/plik.js'. Bez ./ Node szuka w node_modules.

Pokaż różnicę między named a default import. Alias przydatny gdy dwa moduły eksportują tę samą nazwę. Import * - importuje wszystko jako obiekt namespace.

Struktura projektu z modułami


// Typowa struktura projektu testowego:
//
// cypress/
// ├── e2e/
// │   ├── login.cy.js
// │   └── search.cy.js
// ├── pages/
// │   ├── LoginPage.js      ← klasy Page Object
// │   └── SearchPage.js
// ├── fixtures/
// │   └── users.json        ← dane testowe
// └── support/
//     ├── commands.js        ← komendy Cypress
//     └── helpers.js         ← funkcje pomocnicze
Pokaż typową strukturę folderów w projekcie Cypress. Każdy folder ma swoją odpowiedzialność - separacja obaw (separation of concerns).

Obsługa błędów

try / catch / finally / throw

Błędy są normalną częścią programowania. Ważne, żeby umieć je obsługiwać. W testach - obsługa błędów pomaga tworzyć stabilniejsze scenariusze.

try / catch - łapanie błędów


try {
  // Kod, który może rzucić błąd
  const data = JSON.parse("{ nieprawidłowy JSON }");
  console.log(data);
} catch (error) {
  // Obsługa błędu
  console.log("Wystąpił błąd:", error.message);
}
// Program działa dalej!
console.log("To się wykona");
Wystąpił błąd: Expected property name or '}' ...
To się wykona
Bez try/catch błąd zatrzymuje całe wykonanie programu. Z try/catch - łapiemy błąd, obsługujemy go i program kontynuuje.

finally - zawsze się wykona


function fetchData(url) {
  console.log("Rozpoczynam pobieranie...");
  try {
    // próba pobrania danych
    const response = fetch(url);
    console.log("Dane pobrane!");
  } catch (error) {
    console.log("Błąd:", error.message);
  } finally {
    // Wykona się ZAWSZE - i po sukcesie, i po błędzie
    console.log("Czyszczenie zasobów...");
  }
}

Kiedy finally?

Do zamykania połączeń, czyszczenia, logowania - operacji, które muszą się wykonać niezależnie od wyniku.

finally jest opcjonalny, ale bardzo przydatny. Typowe zastosowanie: zamknięcie połączenia z bazą, zatrzymanie loadera, logowanie wyniku.

throw new Error() - rzucanie błędów


function validateAge(age) {
  if (typeof age !== 'number') {
    throw new TypeError("Wiek musi być liczbą");
  }
  if (age < 0 || age > 150) {
    throw new RangeError("Wiek musi być między 0 a 150");
  }
  return true;
}

try {
  validateAge("abc");
} catch (error) {
  console.log(error.name);    // "TypeError"
  console.log(error.message); // "Wiek musi być liczbą"
}
throw pozwala nam tworzyć własne błędy w kodzie. Walidacja danych to typowe zastosowanie - sprawdzamy dane i rzucamy błąd jeśli są nieprawidłowe.

Typy błędów w JavaScript

Typ błędu Kiedy występuje Przykład
TypeError Operacja na złym typie null.toString()
ReferenceError Niezadeklarowana zmienna console.log(xyz)
SyntaxError Błąd składni JSON.parse("{a}")
RangeError Wartość poza zakresem new Array(-1)
Poznanie typów błędów pomaga w debugowaniu. Gdy widzisz TypeError - wiesz, że problem jest z typem danych. W testach najczęściej spotkasz TypeError i ReferenceError.

Error handling w testach


// Testowanie czy funkcja rzuca błąd
function divide(a, b) {
  if (b === 0) throw new Error("Dzielenie przez zero!");
  return a / b;
}

// Test z try/catch
try {
  divide(10, 0);
  console.log("FAIL - oczekiwano błędu");
} catch (error) {
  console.log("PASS - złapano:", error.message);
}

// W Cypress / Jest:
// expect(() => divide(10, 0)).to.throw("Dzielenie przez zero!");

  2. Rzuć Error jeśli brakuje name lub email
  3. Użyj try/catch do przetestowania różnych przypadków
Pokaż, jak try/catch może służyć do testowania - sprawdzamy czy funkcja rzuca błąd. W prawdziwych frameworkach testowych mamy do tego dedykowane asercje. Daj 10 minut na ćwiczenie.

Wprowadzenie do TypeScript

JavaScript z typami

TypeScript to nadzbiór JavaScriptu - każdy kod JS jest prawidłowym kodem TS. TS dodaje system typów, który pomaga wykrywać błędy przed uruchomieniem kodu.

Czym jest TypeScript?

  • Nadzbiór JavaScript - każdy JS jest prawidłowym TS
  • Dodaje statyczne typowanie
  • Kompiluje się do JavaScript
  • Stworzony przez Microsoft
  • Używany w Angular, Cypress, Playwright

// TypeScript
let name: string = "Jan";
let age: number = 30;
let isActive: boolean = true;

// Kompilacja (tsc) ↓

// JavaScript
let name = "Jan";
let age = 30;
let isActive = true;
TypeScript = JavaScript + typy. Przeglądarka nie rozumie TS, więc trzeba go skompilować do JS. Kompilator TypeScript (tsc) sprawdza typy i generuje czysty JavaScript.

Dlaczego TypeScript?

JavaScript


function greet(user) {
  return "Cześć, " + user.name;
}

// Brak błędu do momentu
// uruchomienia!
greet("Jan");
// "Cześć, undefined" 😔

TypeScript


function greet(user: { name: string }) {
  return "Cześć, " + user.name;
}

// BŁĄD w edytorze!
// Argument type 'string' is
// not assignable to parameter
greet("Jan"); // ❌ Błąd kompilacji
  • Bezpieczeństwo - błędy wykrywane przed uruchomieniem
  • Autocomplete - edytor podpowiada właściwości i metody
  • Dokumentacja - typy opisują kształt danych
To kluczowy slajd - pokaż, jak TS chroni przed błędami, które w JS wyjdą dopiero na produkcji. Autocomplete to ogromna oszczędność czasu - edytor zna typ i podpowiada metody.

Instalacja TypeScript

// Kompilacja pliku
npx tsc plik.ts

// Inicjalizacja konfiguracji
npx tsc --init

Typy podstawowe


// Typy proste (primitives)
let productName: string = "Szampon";
let price: number = 12.99;
let isAvailable: boolean = true;
let nothing: null = null;
let notDefined: undefined = undefined;

// any - wyłącza sprawdzanie typów (unikaj!)
let flexible: any = "tekst";
flexible = 42;        // OK
flexible = true;      // OK

// Tablice
let scores: number[] = [10, 20, 30];
let names: string[] = ["Jan", "Anna"];

Unikaj any

any wyłącza sprawdzanie typów - traci się cały sens TypeScript. Używaj tylko w ostateczności.

Typy podstawowe odpowiadają typom JS: string, number, boolean. any to "wyjście awaryjne" - gdy nie znamy typu. Ale tracisz wtedy ochronę TS. Tablice typujemy dodając [] po typie elementu.

Adnotacje typów


// Jawna adnotacja typu
let city: string = "Łódź";

// Inferencja typów - TS sam "zgaduje" typ
let country = "Polska";  // TS wie, że to string
country = 42;            // ❌ Błąd!

// Zmienne bez wartości - adnotacja wymagana
let username: string;
username = "Jan";        // ✅ OK
username = 123;          // ❌ Błąd!

Inferencja typów

Nie musisz pisać typu wszędzie - TypeScript jest sprytny i sam wywnioskuje typ z przypisanej wartości.

TS ma silną inferencję typów - jeśli przypisujesz wartość od razu, TS sam wykrywa typ. Adnotacja jest potrzebna, gdy deklarujesz zmienną bez wartości początkowej. Dobra praktyka: polegaj na inferencji, dodawaj adnotacje gdy potrzebne.

Typy dla funkcji


// Typowanie parametrów i wartości zwracanej
function add(a: number, b: number): number {
  return a + b;
}

// Arrow function z typami
const multiply = (a: number, b: number): number => a * b;

// Funkcja bez wartości zwracanej
function logMessage(message: string): void {
  console.log(message);
}

// Parametry opcjonalne
function greet(name: string, greeting?: string): string {
  return `${greeting || "Cześć"}, ${name}!`;
}

greet("Jan");           // "Cześć, Jan!"
greet("Jan", "Hej");   // "Hej, Jan!"
Typ zwracany po dwukropku za nawiasami parametrów. void = funkcja nic nie zwraca (jak console.log). ? = parametr opcjonalny - może być pominięty. Musi być na końcu listy.

Tablice z typami

Znak ?

Type aliases i Union types


// Type alias - nadanie nazwy typowi
type ID = string | number;

let userId: ID = 123;
userId = "abc-456";  // ✅ OK - string | number

// Union type - jeden z kilku typów
type Status = "active" | "inactive" | "blocked";

let userStatus: Status = "active";    // ✅ OK
userStatus = "deleted";               // ❌ Błąd!

// Łączenie interfejsów
interface Address {
  street: string;
  city: string;
}

type UserWithAddress = User & Address; // intersection
Type alias to alternatywa dla interface - nadaje nazwę złożonemu typowi. Union type (|) = "jeden z tych typów". Literal union = konkretne wartości. Intersection (&) = "oba typy jednocześnie".

Union types w praktyce


// Parametr może być stringiem lub liczbą
function formatId(id: string | number): string {
  if (typeof id === "string") {
    return id.toUpperCase();
  }
  return `ID-${id}`;
}

formatId("abc");   // "ABC"
formatId(123);     // "ID-123"

// Typ odpowiedzi z API
type ApiResponse = {
  success: true;
  data: User[];
} | {
  success: false;
  error: string;
};

function handleResponse(response: ApiResponse) {
  if (response.success) {
    console.log(response.data); // TS wie, że data istnieje
  } else {
    console.log(response.error); // TS wie, że error istnieje
  }
}
Discriminated union (success: true vs false) to potężny wzorzec. TypeScript automatycznie zawęża typ wewnątrz if - "type narrowing". Często używane do modelowania odpowiedzi z API.

enum Category { Cosmetics, Hygiene, Food }

interface Product {
  name: string;
  price: number;
  category: Category;
  discount?: number;

function calculateFinalPrice(product: Product): number {
  const discount = product.discount ?? 0;
  return product.price * (1 - discount / 100);
}

const products: Product[] = [
  { name: "Szampon", price: 15.99, category: Category.Cosmetics },

];

  p => p.category === Category.Cosmetics
);
Omów rozwiązanie. Zwróć uwagę na operator ?? (nullish coalescing) - zwraca prawą stronę jeśli lewa jest null/undefined. Pokaż jak enum zapobiega literówkom w kategorii.

Pytanie 1: Zmienne i typy

Jaki będzie wynik typeof null?

  1. "null"
  2. "undefined"
  3. "object"
  4. "boolean"

Odpowiedź: 3 - "object" (znany bug w JS)

Klasyczne pytanie o historyczny bug w JavaScript. typeof null zwraca "object" - to błąd z pierwszej wersji JS, którego nie można naprawić ze względu na kompatybilność wsteczną.

Pytanie 2: Zmienne

Które stwierdzenie o const jest prawdziwe?

  1. Wartość const nigdy nie może się zmienić
  2. Nie można zmienić referencji, ale można modyfikować obiekt
  3. const działa tak samo jak let
  4. const ma zasięg globalny

Odpowiedź: 2 - Referencja jest stała, ale zawartość obiektu/tablicy można zmieniać

Ważne rozróżnienie: const chroni przed ponownym przypisaniem, ale nie "zamraża" obiektu. const arr = [1,2,3]; arr.push(4) - to działa! Ale arr = [5,6] - błąd.

Pytanie 3: Operatory

Jaki jest wynik "5" === 5?

  1. true
  2. false
  3. "5"
  4. TypeError

Odpowiedź: 2 - === porównuje typ i wartość, string !== number

=== (strict equality) sprawdza zarówno typ jak i wartość. "5" to string, 5 to number - różne typy = false. == (loose equality) skonwertowałoby i zwróciło true - dlatego zawsze używamy ===.

Pytanie 4: Instrukcje warunkowe

Które wartości są falsy w JavaScript?

  1. 0, "", null, undefined, NaN, false
  2. 0, "0", null, undefined
  3. 0, "", [], {}
  4. null, undefined, false

Odpowiedź: 1 - Jest 6 wartości falsy. Uwaga: "0", [], {} są truthy!

6 falsy values - warto zapamiętać. Wszystko inne jest truthy. Częsty błąd: "0" (niepusty string) i [] (pusta tablica) są truthy!

Pytanie 5: Pętle

  2. 3 razy (0, 1, 2)
  3. 4 razy (0, 1, 2, 3)

Odpowiedź: 2 - break przerywa pętlę gdy i === 3, wypisuje: 0, 1, 2

break przerywa pętlę natychmiast - gdy i === 3, console.log się nie wykona. Wypisane zostaną: 0, 1, 2. Czyli 3 iteracje.

Zadanie 3: Co robi ten kod?

Przeczytaj kod i opisz: (a) co robi, (b) jaki będzie wynik:


async function checkStatus(urls) {
  const results = [];

  for (const url of urls) {
    try {
      const response = await fetch(url);
      results.push({
        url,
        status: response.status,
        ok: response.ok
      });
    } catch (error) {
      results.push({
        url,
        status: 0,
        ok: false
      });
    }
  }

  return results.filter(r => !r.ok);
}
Odpowiedź: Funkcja sprawdza listę URLi, pobiera status każdego. Jeśli fetch się nie uda - dodaje obiekt z status: 0. Na końcu filtruje i zwraca TYLKO te, które nie są OK (błędy). To typowa funkcja do health-checku wielu endpointów.

Szybkie podsumowanie - pokaż jak dużo przeszli w 5 dni. Od "czym jest zmienna" do klas, modułów i TypeScript. To duży postęp!

Co dalej? Ścieżka nauki

  • Utrwalaj podstawy - codziennie 15-30 min praktyki
  • Cypress - pisz testy automatyczne w prawdziwych projektach
  • TypeScript - stopniowo dodawaj typy do swoich testów
  • k6 - testy wydajnościowe z JavaScript
  • Node.js - JavaScript po stronie serwera

Klucz do sukcesu

Regularna praktyka jest ważniejsza niż intensywna nauka raz na miesiąc.

Podkreśl, że najważniejsza jest regularna praktyka. 15 minut dziennie > 3h raz w tygodniu. Zachęć do pisania prawdziwych testów w pracy - to najlepsza forma nauki.
