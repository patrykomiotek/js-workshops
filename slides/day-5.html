Testy E2E - Warsztat Praktyczny

Szkolenie JavaScript dla testerów manualnych

Rossmann - Bottega

Ostatni dzień szkolenia - dzień praktyczny! Dziś uczestnicy zastosują wszystko, czego nauczyli się w ciągu ostatnich 4 dni. Główna część dnia to warsztat, w którym wspólnie zbudujemy testy E2E dla aplikacji rossmann-test-app. Zachęć uczestników do aktywnego kodowania.

Agenda

  • Powtórzenie dnia 4 ~30 min
  • Testowanie dynamicznych danych ~1h
  • Dobre praktyki programistyczne w testach ~1h
  • Warsztat - Budujemy testy dla rossmann-test-app ~3h
  • Podsumowanie szkolenia ~30 min
Omów plan dnia. Podkreśl, że dzisiejszy dzień jest w dużej mierze praktyczny. Po krótkim powtórzeniu i dwóch blokach teorii z przykładami, przejdziemy do 3-godzinnego warsztatu. Przerwy będą elastyczne - dostosowane do tempa grupy.

Powtórzenie: Strategie selektorów

Uszereguj selektory od najlepszego do najgorszego w kontekście testów:

  1. cy.get('.btn-primary')
  2. cy.get('[data-testid="submit-btn"]')
  3. cy.get('#app > div:nth-child(3) > button')
  4. cy.get('button').contains('Wyślij')

Odpowiedź: 2, 4, 1, 3 - data-testid jest najbardziej stabilny, zagnieżdżone selektory CSS są najgorsze

Przypomnij piramidę selektorów z dnia 4. data-testid to dedykowany atrybut testowy, nie zmienia się przy refaktoryzacji UI. Tekst przycisku jest dobry, ale może się zmienić przy tłumaczeniach. Klasa CSS może się zmienić przy redesignie. Zagnieżdżone selektory są kruche - każda zmiana w strukturze HTML je psuje.

Powtórzenie: Page Object Model

Co jest główną zaletą wzorca Page Object Model?

  1. Testy działają szybciej
  2. Nie trzeba pisać selektorów
  3. Zmiana UI wymaga aktualizacji tylko w jednym miejscu
  4. Testy automatycznie się naprawiają

Odpowiedź: 3 - POM enkapsuluje selektory i akcje w jednym miejscu. Gdy zmieni się UI, aktualizujemy tylko Page Object.

POM to wzorzec projektowy, nie magiczne rozwiązanie. Wymaga pracy z góry, ale oszczędza ogromną ilość czasu przy utrzymaniu testów. Każda strona/komponent ma swoją klasę z selektorami i metodami akcji.

Powtórzenie: Co automatyzować?

Sklep internetowy Rossmann - który scenariusz jest NAJLEPSZYM kandydatem do automatyzacji?

  1. Sprawdzenie, czy nowy baner wygląda ładnie
  2. Proces zakupowy: dodanie do koszyka - checkout - potwierdzenie
  3. Jednorazowa migracja danych z innego systemu
  4. Testowanie animacji po najechaniu na produkt

Odpowiedź: 2 - Krytyczny, powtarzalny, stabilny scenariusz biznesowy - idealny kandydat do automatyzacji

Przypomnij kryteria: wysoka wartość biznesowa, powtarzalność, stabilność, możliwość weryfikacji. Baner i animacje to testy wizualne (inne narzędzia). Jednorazowa migracja nie wymaga automatyzacji. Proces zakupowy to rdzeń biznesu - musi działać zawsze.
01

Testowanie dynamicznych danych

Jak pisać stabilne testy, gdy dane się zmieniają?

To jeden z największych wyzwań w testach E2E. Dane dynamiczne to: daty, losowe ID, spersonalizowane treści, ceny po promocjach, rekomendacje. Nauczymy się strategii radzenia sobie z tymi problemami.

Problem: dane się zmieniają

Typowe źródła niestabilnych testów:

  • Daty i godziny - "Dziś jest 07.03.2026"
  • Losowe identyfikatory - "ORD-48291"
  • Spersonalizowane treści - rekomendacje, oferty
  • Ceny i promocje - zmieniają się w czasie
  • Dane z zewnętrznych API - mogą być niedostępne

Uwaga

Test, który przechodzi dzisiaj, a nie przechodzi jutro - to nie jest test, to loteria.

Poproś uczestników o przykłady z ich codziennej pracy - jakie dane się zmieniają w aplikacjach, które testują? To pomoże im powiązać teorię z praktyką.

Daty i czasy - kruche asercje

Źle - kruche


// Cypress - nie rób tak!
cy.get('[data-testid="date"]')
  .should('have.text', '07.03.2026');

// Playwright - nie rób tak!
await expect(greeting).toHaveText(
  'Witaj, Anna! Dzis jest 07.03.2026'
);

Dobrze - elastyczne


// Cypress - regex na format daty
cy.get('[data-testid="date"]')
  .invoke('text')
  .should('match', /\d{2}\.\d{2}\.\d{4}/);

// Playwright - partial match
await expect(greeting)
  .toContainText('Witaj, Anna!');
Pokaż na żywo: uruchom test z hardkodowaną datą - przejdzie dziś, jutro nie. Strategia: sprawdzaj format daty, nie konkretną wartość. Albo sprawdzaj tylko część tekstu, która jest stała.

Strategia: zakresy dat i czasów


// Cypress - sprawdź timestamp w rozsądnym zakresie
cy.get('[data-testid="order-time"]')
  .invoke('attr', 'data-timestamp')
  .then((timestamp) => {
    const orderTime = new Date(Number(timestamp));
    const now = new Date();
    const diffMinutes = (now - orderTime) / 1000 / 60;
    expect(diffMinutes).to.be.lessThan(5);
  });
    

// Playwright - sprawdź, że data jest "dzisiejsza"
test('should display today date', async ({ page }) => {
  await page.goto('/dashboard');

  const dateText = await page.getByTestId('current-date').textContent();
  const today = new Date();
  const expectedDay = String(today.getDate()).padStart(2, '0');
  const expectedMonth = String(today.getMonth() + 1).padStart(2, '0');

  expect(dateText).toContain(`${expectedDay}.${expectedMonth}`);
});
Dwa podejścia: generuj oczekiwaną datę dynamicznie albo sprawdzaj zakres. Unikaj hardkodowania dat - to najczęstsza przyczyna "zepsutych" testów w poniedziałek rano.

Losowe identyfikatory


// Cypress - sprawdzaj format, nie wartość
cy.get('[data-testid="order-id"]')
  .invoke('text')
  .should('match', /^ORD-\d+$/);

// Playwright - sprawdzaj format
const orderId = await page.getByTestId('order-id').textContent();
expect(orderId).toMatch(/^ORD-\d+$/);

Regex w testach

/^ORD-\d+$/ - zaczyna się od "ORD-", potem jedna lub więcej cyfr. Sprawdzamy format, nie wartość.


// Inne przydatne wzorce regex:
/^[A-Z]{2}-\d{6}$/       // np. PL-123456
/^\w+@\w+\.\w+$/         // prosty email
/^\d{2}-\d{3}$/           // kod pocztowy
/^[0-9a-f]{8}-[0-9a-f]{4}/ // UUID (początek)
Wyjaśnij regex krok po kroku: ^ - początek, ORD- - literalny tekst, \d+ - cyfry, $ - koniec. Nie trzeba znać całego regex - wystarczy kilka podstawowych wzorców. Pokaż regex101.com jako narzędzie do testowania wyrażeń regularnych.

Spersonalizowane treści

Rekomendacje, oferty, propozycje - różne dla każdego użytkownika


// Źle - konkretny produkt może zniknąć z rekomendacji
cy.get('[data-testid="recommendations"]')
  .should('contain', 'Szampon XYZ');

// Dobrze - sprawdzaj strukturę, nie treść
cy.get('[data-testid="recommendations"]')
  .should('exist')
  .find('[data-testid="product-card"]')
  .should('have.length.greaterThan', 0);

// Dobrze - sprawdzaj, że każda karta ma wymagane elementy
cy.get('[data-testid="product-card"]').first().within(() => {
  cy.get('[data-testid="product-name"]').should('not.be.empty');
  cy.get('[data-testid="product-price"]').should('not.be.empty');
  cy.get('[data-testid="add-to-cart"]').should('be.visible');
});
Klucz: testuj STRUKTURĘ, nie TREŚĆ dynamiczną. Sprawdź, że sekcja rekomendacji istnieje, ma elementy, każdy element ma nazwę, cenę i przycisk. Nie sprawdzaj, JAKI konkretnie produkt jest wyświetlany.

Ceny i formatowanie walutowe


// Sprawdzaj format ceny, nie konkretną wartość
cy.get('[data-testid="product-price"]')
  .invoke('text')
  .should('match', /^\d+,\d{2}\s?zl$/);

// Playwright - sprawdzaj, że cena jest liczbą > 0
const priceText = await page.getByTestId('product-price').textContent();
const price = parseFloat(
  priceText.replace(',', '.').replace(/[^\d.]/g, '')
);
expect(price).toBeGreaterThan(0);

Kiedy sprawdzać konkretną cenę?

Tylko gdy kontrolujesz dane testowe - np. przez mocki API lub fixtures.

Ceny mogą się zmieniać w wyniku promocji, przeliczania walut itp. Lepiej sprawdzać format i zakres wartości niż konkretną cenę. Wyjątek: gdy mockujemy API - wtedy znamy dane.

Mockowanie API - kontrola nad danymi

Zamiast testować z prawdziwym API, przechwytujemy żądania i zwracamy kontrolowane dane

Zalety

  • Kontrolowane dane testowe
  • Testy nie zależą od backendu
  • Szybsze wykonanie
  • Można testować edge cases

Wady

  • Nie testujemy prawdziwego API
  • Mock może się rozjechać z realnym API
  • Wymaga utrzymania mocków
Wyjaśnij analogię: mock to jak dublerka w filmie. Wygląda jak prawdziwy aktor, ale kontrolujemy co robi. Testy z mockami są stabilne, ale nie zastąpią testów integracyjnych. Idealna strategia: większość testów z mockami + kilka smoke testów z prawdziwym API.

Cypress - cy.intercept()


describe('Products page', () => {
  it('displays mocked products', () => {
    cy.intercept('GET', '/api/products', {
      body: [
        { id: 1, name: 'Test Product', price: 29.99 },
        { id: 2, name: 'Another Product', price: 49.99 }
      ]
    }).as('getProducts');

    cy.visit('/products');
    cy.wait('@getProducts');

    cy.get('[data-testid="product-card"]')
      .should('have.length', 2);
    cy.contains('Test Product').should('be.visible');
    cy.contains('29,99 zl').should('be.visible');
  });
});
cy.intercept() przechwytuje żądania HTTP pasujące do wzorca URL. .as() nadaje alias, którego możemy użyć w cy.wait(). cy.wait('@getProducts') czeka na wykonanie żądania. Dzięki temu testy są deterministyczne.

Playwright - page.route()


import { test, expect } from '@playwright/test';

test('displays mocked products', async ({ page }) => {
  // Przechwytujemy żądanie do API
  await page.route('**/api/products', route => {
    route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([
        { id: 1, name: 'Test Product', price: 29.99 },
        { id: 2, name: 'Another Product', price: 49.99 }
      ])
    });
  });

  await page.goto('/products');

  // Teraz wiemy dokładnie, jakie dane zobaczymy
  await expect(page.getByText('Test Product')).toBeVisible();
  await expect(page.getByText('29,99 zl')).toBeVisible();
});
W Playwright odpowiednikiem cy.intercept() jest page.route(). Używamy ** jako wildcard dla dowolnego hosta. route.fulfill() zwraca kontrolowaną odpowiedź. Dzięki temu test nie zależy od stanu bazy danych ani dostępności backendu.

Kiedy mockować, kiedy nie?

Mockuj gdy:

  • Dane są dynamiczne/losowe
  • Backend jest niestabilny
  • Testujesz edge cases (błędy API)
  • Chcesz szybkie testy w CI
  • Zależy Ci na izolacji frontendowej

Nie mockuj gdy:

  • Smoke testy produkcyjne
  • Testy integracyjne API
  • Chcesz przetestować cały flow
  • Sprawdzasz wydajność

Złota zasada

80% testów z mockami (szybkie, stabilne) + 20% bez mocków (realistyczne).

To nie jest albo-albo. Najlepsza strategia to mieszanka. Większość testów E2E z mockami, a kilka kluczowych scenariuszy (happy path) z prawdziwym backendem. Smoke testy na produkcji powinny zawsze używać prawdziwego API.

Mockowanie błędów API


// Cypress - symulacja błędu 500
cy.intercept('GET', '/api/products', {
  statusCode: 500,
  body: { error: 'Server Error' }
}).as('serverError');

cy.visit('/products');
cy.wait('@serverError');
cy.get('[data-testid="error-message"]')
  .should('contain', 'Wystąpił błąd');
    

// Playwright - symulacja błędu serwera
await page.route('**/api/products', route => {
  route.fulfill({ status: 500, body: 'Internal Server Error' });
});
await page.goto('/products');
await expect(page.getByText('Wystąpił błąd')).toBeVisible();

Testuj obsługę błędów

Możesz symulować różne kody: 400, 401, 403, 404, 500 - a nawet timeout sieciowy.

Testowanie obsługi błędów jest często pomijane, a to kluczowe dla UX. Czy aplikacja wyświetla przyjazny komunikat? Czy można spróbować ponownie? Mocki pozwalają łatwo symulować różne kody błędów.

02

Dobre praktyki w testach

Jak pisać testy, które łatwo utrzymać i zrozumieć

Ten blok to zbiór najważniejszych praktyk. Nie trzeba stosować wszystkich od razu - najważniejsze to zacząć od dobrej struktury i nazewnictwa, a resztę dodawać stopniowo.

DRY w testach - z umiarem

Źle - kopiuj/wklej


// Ten sam login w każdym teście
it('test 1', () => {
  cy.visit('/login');
  cy.get('[data-testid="email"]')
    .type('test@rossmann.pl');
  cy.get('[data-testid="password"]')
    .type('Test1234');
  cy.get('[data-testid="login-btn"]')
    .click();
  // ...reszta testu
});

it('test 2', () => {
  // Ten sam login skopiowany...
});

Ale nie przesadzaj!

Czytelność > DRY w testach


// Za dużo abstrakcji
it('test 1', () => {
  doEverything('scenario-1');
});

// Lepiej - czytelne kroki
it('test 1', () => {
  login('test@rossmann.pl');
  addToCart('Szampon');
  checkout();
  verifyOrder();
});
Test powinien być czytelny jak historia. Każdy powinien szybko zrozumieć CO test robi, nawet bez czytania implementacji helperów. Za dużo abstrakcji sprawia, że trzeba przekopywać się przez wiele plików, żeby zrozumieć prosty test.

Custom commands - Cypress


// cypress/support/commands.js
Cypress.Commands.add('login', (email, password) => {
  cy.visit('/login');
  cy.get('[data-testid="email"]').type(email);
  cy.get('[data-testid="password"]').type(password);
  cy.get('[data-testid="login-btn"]').click();
  cy.url().should('include', '/dashboard');
});

Cypress.Commands.add('addToCart', (productName) => {
  cy.contains('[data-testid="product-card"]', productName)
    .find('[data-testid="add-to-cart"]')
    .click();
});
    

// Użycie w teście - czyste i czytelne
it('should complete purchase flow', () => {
  cy.login('test@rossmann.pl', 'Test1234');
  cy.addToCart('Szampon');
  cy.get('[data-testid="cart-badge"]').should('contain', '1');
});
Custom commands to odpowiednik helperów w Cypress. Definiujemy je raz w support/commands.js, a potem używamy w każdym teście. Cypress automatycznie ładuje pliki z katalogu support/.

W Playwright: Fixtures

Odpowiednik Custom Commands z poprzedniego slajdu - w Playwright mechanizm dependency injection.


// fixtures.ts
import { test as base } from '@playwright/test';
import { LoginPage } from './pages/LoginPage';
import { DashboardPage } from './pages/DashboardPage';

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  dashboardPage: async ({ page }, use) => {
    await use(new DashboardPage(page));
  },
});

export { expect } from '@playwright/test';
    

// Użycie w teście
import { test, expect } from './fixtures';

test('should login and see dashboard', async ({ loginPage, dashboardPage }) => {
  await loginPage.login('test@rossmann.pl', 'Test1234');
  await expect(dashboardPage.welcomeMessage).toBeVisible();
});
Fixtures w Playwright to mechanizm dependency injection. Deklarujemy zależności testu, a Playwright automatycznie je tworzy i niszczy. To mocniejszy mechanizm niż beforeEach - fixtures mogą mieć zależności między sobą i automatyczne czyszczenie.

Setup i teardown - hooki


describe('Dashboard', () => {
  beforeEach(() => {
    // Wykonuje się PRZED KAŻDYM testem
    cy.login('test@rossmann.pl', 'Test1234');
  });

  afterEach(() => {
    // Wykonuje się PO KAŻDYM teście
    cy.clearCookies();
    cy.clearLocalStorage();
  });

  it('should display welcome message', () => {
    cy.get('[data-testid="welcome"]').should('be.visible');
  });

  it('should show product list', () => {
    cy.get('[data-testid="product-card"]')
      .should('have.length.greaterThan', 0);
  });
});
beforeEach to idealne miejsce na powtarzalne kroki setup - logowanie, nawigacja, ustawienie stanu. afterEach to czyszczenie - ciasteczka, localStorage, dane testowe. Są też beforeAll/afterAll (before/after w Cypress) - wykonują się raz dla całego describe.

Izolacja testów

Każdy test MUSI być niezależny

Testy nie mogą zależeć od kolejności wykonania ani od stanu zostawionego przez inny test.

Źle


// Test 1 tworzy produkt
it('creates product', () => {
  cy.createProduct('Szampon');
});

// Test 2 ZAKLADA, ze produkt istnieje
it('edits product', () => {
  cy.get('[data-testid="Szampon"]')
    .click(); // Nie zadziala solo!
});

Dobrze


// Każdy test tworzy własne dane
it('edits product', () => {
  // Arrange
  cy.createProduct('Szampon');

  // Act
  cy.get('[data-testid="Szampon"]')
    .click();
  cy.get('[data-testid="name"]')
    .clear().type('Nowy Szampon');
  cy.get('[data-testid="save"]').click();

  // Assert
  cy.contains('Nowy Szampon');
});
Testy mogą być uruchamiane w dowolnej kolejności, równolegle, lub pojedynczo. Jeśli test 2 wymaga danych z testu 1, to jest błąd projektowy. Każdy test powinien sam tworzyć potrzebne dane (lub używać mocków) i sprzątać po sobie.

Wzorzec Arrange-Act-Assert


it('should add product to cart and update total', () => {
  // ARRANGE - przygotuj warunki
  cy.intercept('GET', '/api/products', {
    body: [{ id: 1, name: 'Szampon', price: 15.99 }]
  });
  cy.visit('/products');

  // ACT - wykonaj akcje
  cy.get('[data-testid="add-to-cart"]').first().click();

  // ASSERT - sprawdz wynik
  cy.get('[data-testid="cart-badge"]').should('contain', '1');
  cy.get('[data-testid="cart-total"]').should('contain', '15,99');
});

AAA - czytelna struktura

Arrange (Przygotuj) - Act (Wykonaj) - Assert (Sprawdź). Każdy test powinien mieć te 3 fazy.

AAA to standard w testowaniu. Pomaga utrzymać testy czytelne i dobrze zorganizowane. Arrange: dane, mocki, nawigacja. Act: kliknięcie, wpisanie tekstu, wysłanie formularza. Assert: sprawdzenie wyniku. Nie mieszaj tych faz!

  });

    // ...
  });

    // ...
  });

Dobra nazwa testu to dokumentacja

Czytając nazwy testów powinno się rozumieć, co aplikacja robi - bez czytania kodu.

Dobre nazwy testów to jak dokumentacja wymagań. Gdy test się wysypie w CI, z nazwy powinno być jasne, co nie działa. "test 1", "login test" to złe nazwy. "should redirect to dashboard when credentials are valid" - od razu wiadomo, co się stało.

Dane testowe w plikach JSON


// fixtures/users.json
{
  "validUser": {
    "email": "test@rossmann.pl",
    "password": "Test1234",
    "name": "Anna Testowa"
  },
  "invalidUser": {
    "email": "bad@rossmann.pl",
    "password": "wrong"
  }
}
    

// Cypress - automatyczne ładowanie fixtures
it('should login with valid credentials', () => {
  cy.fixture('users').then((users) => {
    cy.login(users.validUser.email, users.validUser.password);
    cy.contains(users.validUser.name);
  });
});

// Playwright - import JSON
import users from './fixtures/users.json';

test('should login', async ({ page }) => {
  await loginPage.login(users.validUser.email, users.validUser.password);
});
Dane testowe w osobnych plikach JSON. Łatwo zmieniać, łatwo przeglądać, czytelne w kodzie. W Cypress fixtures ładujemy przez cy.fixture(). W Playwright importujemy jak zwykły moduł. Nie hardkoduj danych w testach - wyciągnij do fixtures.

Automatyzacja testów

Testy E2E powinny uruchamiać się automatycznie przy każdym pushu i pull requeście.

GitHub Actions to jedno z wielu narzędzi CI - jest też GitLab CI, Jenkins, Azure DevOps. Kluczowe: testy uruchamiają się automatycznie, wyniki są widoczne w PR, raporty zapisywane jako artefakty. upload-artifact zachowuje raport HTML nawet gdy testy failują (if: failure()).

// playwright.config.ts
export default defineConfig({
  reporter: [
    ['html', { open: 'never' }],
    ['list']
  ],
  use: {
    screenshot: 'only-on-failure',

Screenshoty i video na failurze to must-have. Trace w Playwright nagrywa pełną interakcję, którą można odtworzyć krok po kroku. To nieocenione przy debugowaniu testów, które failują w CI, ale przechodzą lokalnie.

Anty-wzorce w testach E2E

Unikaj tych błędów

  • Hardkodowane waity - cy.wait(5000) zamiast czekania na element
  • Zależne testy - test B wymaga danych z testu A
  • Testowanie implementacji - sprawdzanie klas CSS zamiast zachowania
  • Za dużo w jednym teście - test sprawdza 20 rzeczy na raz
  • Brak cleanup - dane testowe zostają w bazie
  • Ignorowanie flaky testów - "czasem nie przechodzi, to skip"
Omów każdy anty-wzorzec z przykładem. Hardkodowane waity to najczęstszy problem - spowalniają testy i nadal mogą failować na wolnym CI. Flaky testy to tykająca bomba - ignorowanie ich prowadzi do utraty zaufania w cały suite testowy.

Ćwiczenie: Refaktoryzacja testu

Popraw poniższy test stosując poznane praktyki:


it('test', () => {
  cy.visit('/login');
  cy.get('#email').type('test@rossmann.pl');
  cy.get('#pass').type('Test1234');
  cy.get('.submit').click();
  cy.wait(3000);
  cy.get('.products .item:nth-child(1) .add-btn').click();
  cy.wait(2000);
  cy.get('.cart-count').should('have.text', '1');
  cy.get('.products .item:nth-child(1) .price')
    .should('have.text', '15.99 zl');
});

Co poprawić: nazewnictwo, selektory, waity, strukturę AAA, DRY

Czas: 15 minut

Daj uczestnikom 15 minut. Główne problemy do znalezienia: zła nazwa testu, selektory CSS zamiast data-testid, cy.wait(3000) zamiast czekania na element, za dużo w jednym teście, brak struktury AAA. Po czasie pokaż rozwiązanie na następnym slajdzie.

Rozwiązanie - po refaktoryzacji


describe('Cart', () => {
  beforeEach(() => {
    cy.login('test@rossmann.pl', 'Test1234');
  });

  it('should update cart badge when product is added', () => {
    // Arrange
    cy.get('[data-testid="product-card"]').first().as('product');

      .find('[data-testid="add-to-cart"]')

Co się zmieniło?

Nazwa opisuje zachowanie, selektory data-testid, brak hardkodowanych waitów, login w beforeEach, wzorzec AAA.

Omów każdą zmianę. Login wyciągnięty do beforeEach. Nazwa testu opisuje oczekiwane zachowanie. Selektory data-testid zamiast klas CSS. Brak cy.wait() - Cypress automatycznie czeka na elementy. Test sprawdza jedną rzecz - badge koszyka po dodaniu produktu.

Podpowiedzi - loginPage

Potrzebne selektory i metody:

  • cy.get('[data-testid="login-email-input"]') - pole email
  • cy.get('[data-testid="login-password-input"]') - pole hasło

Odkrywaj fragmenty jeden po drugim. Daj uczestnikom chwilę na każdy selektor. Pytaj: "Jaką metodę powinien mieć loginPage?" - niech sami wymyślą interfejs obiektu.

Rozwiązanie - loginPage.js


// cypress/support/pages/loginPage.js
export const loginPage = {
  visit() {
    cy.visit('/');
  },

  fillEmail(email) {
    cy.get('[data-testid="login-email-input"]').type(email);
  },

  fillPassword(password) {
    cy.get('[data-testid="login-password-input"]').type(password);
  },

  submit() {
    cy.get('[data-testid="login-submit-btn"]').click();
  },

  login(email, password) {
    this.visit();
    this.fillEmail(email);
    this.fillPassword(password);
    this.submit();
  },

  getError() {
    return cy.get('[data-testid="login-error"]');
  }
};
W Cypress Page Object to zwykły obiekt z metodami, nie klasa. Każda metoda używa cy.get() z data-testid. Metoda login() łączy visit + wypełnienie formularza + submit. getError() zwraca element - można na nim wywołać asercje.

Rozwiązanie - login.cy.js


// cypress/e2e/login.cy.js
import { loginPage } from '../support/pages/loginPage';

describe('Login Page', () => {
  beforeEach(() => {
    loginPage.visit();
  });

  it('should redirect to dashboard on valid login', () => {
    cy.fixture('users').then((users) => {
      loginPage.login(users.validUser.email, users.validUser.password);
      cy.url().should('include', 'dashboard');
      cy.contains(users.validUser.name).should('be.visible');
    });
  });

  it('should show error on invalid credentials', () => {
    cy.fixture('users').then((users) => {
      loginPage.login(users.invalidUser.email, users.invalidUser.password);
      loginPage.getError().should('be.visible');
      loginPage.getError().should('contain', 'Nieprawidlowy');
    });
  });

  it('should clear session on logout', () => {
    cy.fixture('users').then((users) => {
      loginPage.login(users.validUser.email, users.validUser.password);
      cy.url().should('include', 'dashboard');
      cy.get('[data-testid="logout-btn"]').click();
      cy.url().should('include', 'index');
    });
  });
});
Omów test po teście. Zwróć uwagę na beforeEach - nawigujemy do strony logowania. cy.fixture('users') ładuje dane z cypress/fixtures/users.json. Test 1: logowanie + sprawdzenie URL i nazwy użytkownika. Test 2: zły login + komunikat błędu. Test 3: login, logout, sprawdzenie powrotu do strony głównej. Każdy test jest niezależny dzięki beforeEach.

Rozwiązanie - dashboardPage.js


// cypress/support/pages/dashboardPage.js
export const dashboardPage = {
  getProductCards() {
    return cy.get('[data-testid="product-card"]');
  },

  search(query) {
    cy.get('[data-testid="search-input"]').clear().type(query);
  },

  filterByCategory(category) {
    cy.get(`[data-testid="category-${category}"]`).click();
  },

  addFirstProductToCart() {
    cy.get('[data-testid="product-card"]').first()
      .find('[data-testid="add-to-cart"]').click();
  },

  getCartBadge() {
    return cy.get('[data-testid="cart-badge"]');
  }
};
W Cypress Page Object to obiekt z metodami zwracającymi elementy. getProductCards() zwraca wszystkie karty produktów. search() czyści pole i wpisuje tekst. filterByCategory() klika odpowiedni filtr. addFirstProductToCart() klika przycisk na pierwszej karcie.

// cypress/support/pages/cartPage.js
export const cartPage = {
  visit() {
    cy.visit('/cart.html');
  },

  getCartItems() {
    return cy.get('[data-testid="cart-item"]');
  },

  getTotal() {
    return cy.get('[data-testid="cart-total"]');
  },

  getEmptyMessage() {
    return cy.get('[data-testid="empty-cart"]');
  },

  changeQuantity(index, qty) {
    cy.get('[data-testid="cart-item"]').eq(index)
      .find('[data-testid="quantity-input"]').clear().type(qty);

      .find('[data-testid="remove-btn"]').click();

cartPage jest prostsza niż dashboardPage. Kluczowe metody: changeQuantity przyjmuje indeks elementu i nową ilość, removeItem usuwa element po indeksie. getTotal zwraca element sumy na którym można wywołać asercje.

// cypress/e2e/cart.cy.js
import { loginPage } from '../support/pages/loginPage';
import { dashboardPage } from '../support/pages/dashboardPage';
import { cartPage } from '../support/pages/cartPage';

describe('Cart Page', () => {
  beforeEach(() => {
    cy.fixture('users').then((users) => {
      loginPage.login(users.validUser.email, users.validUser.password);
    });
    dashboardPage.addFirstProductToCart();
    cartPage.visit();
  });

  it('should display added products in cart', () => {
    cartPage.getCartItems().should('have.length', 1);
  });

  it('should update quantity of item', () => {
    cartPage.changeQuantity(0, '3');
    // Poczekaj na aktualizację sumy
    cartPage.getTotal().should('not.contain', '0');
  });
beforeEach: logowanie + dodanie produktu + nawigacja do koszyka. Każdy test zaczyna z jednym produktem w koszyku. Test ilości: zmiana na 3, sprawdzenie że suma się zmieniła.

    .and('contain', 'Dodano do koszyka');
});
Toast: sprawdzamy, że pojawia się i ma odpowiedni tekst. Cypress automatycznie retryuje asercje, więc nie trzeba czekać na animację. cy.login() to custom command zdefiniowany w cypress/support/commands.js.

Rozwiązanie - testy zaawansowane (2/2)


// Test responsywnosci - mobilny viewport
it('should adapt to mobile viewport', () => {
  cy.viewport(390, 844);
  cy.login('test@rossmann.pl', 'Test1234');

  // Menu mobilne powinno być widoczne
  cy.get('[data-testid="mobile-menu"]').should('be.visible');
  // Nawigacja desktopowa powinna być ukryta
  cy.get('[data-testid="desktop-nav"]').should('not.be.visible');
});

// Alternatywa - użyj presetów Cypress
it('should work on iPhone viewport', () => {
  cy.viewport('iphone-xr');
  cy.login('test@rossmann.pl', 'Test1234');
  cy.get('[data-testid="mobile-menu"]').should('be.visible');
});
Dwa podejścia do testowania responsywności. cy.viewport(width, height) daje pełną kontrolę. cy.viewport('iphone-xr') używa wbudowanych presetów urządzeń. Cypress ma wiele presetów: iphone-6, iphone-xr, samsung-s10, macbook-15 itp.

Typowe problemy i rozwiązania

FAQ z warsztatu

  • "Test timeout" - element nie istnieje lub ma inny data-testid. Sprawdź w DevTools.
  • "Element not visible" - element jest w DOM, ale ukryty. Sprawdź CSS display/visibility.
  • "cy.get() returned more than one element" - selektor pasuje do wielu elementów. Użyj .first(), .eq(0) lub bardziej precyzyjnego selektora.
  • "Timed out retrying" - Cypress nie może znaleźć elementu w czasie defaultCommandTimeout.

Debugowanie

Użyj npx cypress open do interaktywnego debugowania testów. Cypress pokazuje każdy krok z podglądem DOM.

Przejdź przez typowe problemy, które uczestnicy prawdopodobnie napotkali. Pokaż jak debugować każdy z nich. Cypress open mode jest bardzo pomocny - pokazuje każdy krok testu z podglądem aplikacji i snapshotem DOM. Można też użyć cy.pause() do zatrzymania testu.
